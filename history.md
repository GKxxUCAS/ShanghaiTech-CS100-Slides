# 关于 Spring 2022 - Spring 2024 CS100 的情况的总结

Oct. 2025

我自 Spring 2022 第一次担任 CS100 助教开始，尝试对这门课程进行了各种改革。 2024 年我从上科大毕业，那时我曾想过要彻底翻页，不再花时间做有关 CS100 的事情，但是做事还是应该善始善终——正如我在上科大乒乓球社做社长时，尽管工作做得不好，也在最后一年和校队的朋友们共同完成了交接工作。过去的一年里 CS100 在新的助教的带领下也完成得不错，尽管出现了一些风波。再三思考后，我觉得应该把我过去所做的工作做一个总结，特别是把很多内容我在设计时的思考写下来，以便于后来的助教理解。

## 课程设定

注：以下内容仅限于 2022-2024 年，后续课程安排可能有变化。

CS100 （“计算机编程”）是主要面向 CS 、 EE 、 BME 、数学所的本科生的编程基础课，其中 CS 学生普遍在大一下的春学期修读。自 2023 年起，秋学期也开设 CS100 ，主要供非 CS 的学生修读，以方便学生选课，同时也分担春学期的部分压力。通常春学期有 250+ 人选课，秋学期选课人数较少， Fall 2023 的选课人数不到 50 。

CS100 的课程内容为 C 和 C++ 编程，其中前六周是 C ，后十周是 C++ ，每周有 4 个课时，同时还有 2 课时的习题课。习题课以讲课为主，包含对正课内容的复习和补充。有 1-2 次 quiz ，通常在 C 的部分会有一次 quiz ， C++ 的部分可能有第二次，视进度情况而定。有 1 次考试，在第 13-14 周（Spring 2023 由于时间安排不当拖到了 15 周），考查范围包含 C 的全部内容和 C++ 的 class 基础、拷贝控制（包括移动）、重载运算符、智能指针、 STL 基础（迭代器和算法）、继承与多态初步，包含习题课上的内容。根据 Spring 2024 的内容顺序，即考到 Lecture 22 （含）。作业有 8 次左右，通常前 3 - 4 次是 C 的作业，后面是 C++ 的作业。最终的成绩组成为作业 72% + 考试 22% + quiz 4% + 习题课签到 2% ，其中作业的 72% 的具体分配视具体情况而定，通常综合性较强的作业占比应稍高一些， C++ 的作业占比可能高于 C ，例如 Spring 2024 的各次作业占比为 7% + 7% + 7% + 11% + 7% + 11% + 11% + 11% 。 Spring 2022 由于疫情原因，取消了考试和 quiz 并把作业改为 10 次。

Spring 2022 有大约 15 个习题班，每个班 20 人左右，由 1 位助教负责。从 Spring 2023 开始，改为 7-8 个习题班，每个班 40 人左右，由 2-3 位助教负责，其中应有 1 位水平较高的助教。

上科大还一直有查重的传统：只要两个人的代码被判定为抄袭，不管是谁抄谁，双方都是同等处罚，除非这种抄袭是通过极其恶劣而无法预防的手段做到的，比如盗取对方的 OJ 账号密码。保护自己的代码免于被抄的风险也是学生必须要做的，比如人离开电脑时要锁定，不要把自己的代码上传到公开平台等等。具体执行的时候不同的老师/助教会有不同的政策，比如视具体情况酌情处理，被抄的人从轻处理，初犯从轻处理等，很多课程还会有一个 argue 的环节，会把学生叫过来尝试让他证明自己有能力写出这份代码，包括但不限于解释代码中的具体细节。处罚的力度上，有的会把判重的那道题置 0 分，有的会把当次作业记 0 分，有的会倒扣分（以减小抄袭的数学期望，避免“稳赚不赔”），有的还会对“承认”和“不承认”区别处罚力度。我也听说有些特别变态的一道题判重就直接 Fail 的故事。总的来说上科大在课程方面对学术诚信是比较重视的，尤其是考虑到在很多课程中作业都在最终成绩乃至于 GPA 里有着实打实的占比，特别像 CS100 作业占比 72% ，这和国内多数传统高校占 40% 左右的、可能相对模糊一些的“平时成绩”概念不同。此外，有的课程允许学生帮别人 debug ，前提是你真的是在 debug 而不是直接帮别人按自己的做法写一份。

这些是课程的基本情况，我所做的各种改变也始终是在这个大的框架之下进行的。对于这个框架本身实际上我也有一些改动的想法，但由于时间和能力问题并没有实施，这些会在后面谈到。

## Spring 2022 ：问题，和一些初步的尝试

### 问题

最初在 2022 年春学期做这门课的助教时，我对于课程内容的最直接的感受就是**不够专业**，这主要体现在以下几个方面：
1. 对 C 和 C++ 的 ISO 标准的忽视。有一些内容直接是不符合标准的，比如说 `operator+=` 的重载可以声明为 non-member ；有一些内容没有用标准的术语来讨论，致使讨论没有明确的框架和边界（例如，不是说“未定义的行为”，而是说“它会输出垃圾值”或者“会造成内存中的其它位置的数据被修改”云云）；还有一些内容在不同的语言标准中的区别没有被指出，比如除法的舍入规则。
2. 对代码风格、规范性、最佳实践的忽视。 C 和 C++ ，前者是非常简陋的，后者是非常复杂的，而它们都是非常灵活的，都是直接接触底层系统而没有间接的虚拟环境或解释器保护的语言。想在 C 和 C++ 里实现一个功能往往有很多种不同的办法，而这些办法看似都正确，实则某些相比于另一些是更容易出错的、有隐患的，或者是可读性更差的、不易维护的。在这一话题上，有著作如《C Traps and Pitfalls》《Ruminations on C++》以及 Scott Meyers 的“Effective”系列，有被业界采纳的“安全编程子集”如 [C++ Core Guidelines](https://isocpp.github.io/CppCoreGuidelines/CppCoreGuidelines) 、 CERT [C](https://wiki.sei.cmu.edu/confluence/display/c)/[C++](https://wiki.sei.cmu.edu/confluence/pages/viewpage.action?pageId=88046682) Coding Standard ，有相关检测工具如 [Clang-Tidy](https://clang.llvm.org/extra/clang-tidy/) ，在教学方面也有很多[相关的论文](https://arxiv.org/abs/2502.11230)提出工具或分享经验，这些足以证明在编程教学中引入这些讨论的必要性和可行性。
3. 作业带有浓重的算法竞赛气息。只进行运行时的测试，只给出“Wrong Answer”、“Runtime Error”、“Accepted”之类的反馈而没有更进一步的指引。作业题目对知识点的考察浅尝辄止，也缺乏综合性，题目的素材有时显得过于刻意、没活硬整。我理想中的作业题目的素材背景应该来自真实的世界，比如对电梯、进站闸机、记分牌等事物的状态机建模（而不是出题人自己生造一些逻辑来让学生模拟），或者引导学生探索一些在本课程以外的计算机世界，比如引入一点点设计模式。

这种专业性的缺乏是体现在课程的方方面面的，包括考试，包括老师和助教。当时助教的任务主要是讲习题课（包括做习题课课件），并且包括我在内的三位助教负责出作业。本来可能还要出 quiz 和考试卷，但由于疫情原因取消了。于是，我首先考虑在我的职责范围内做一些改变，也就是习题课和作业。

### 初步的改进尝试

习题课方面，当时老师定的规则是每位助教负责一周的内容，大家轮流准备，并且可以借助往年的材料。不过我自己按我的想法重做了习题课的全部课件，力求在习题课上重新建立起对标准的认识和对代码规范性与最佳实践的强调，不过那时我的参考资料主要是我以前读过的各种书，比如我在初三时就读完的《C++ Primer》以及后来读的“Effective”系列、《Ruminations on C++》等等，这些书尽管都出自足够权威的人之手，但也并非标准原文，难免还是有个别不严谨和过时的地方。此外，从第 7 次习题课开始我把我的习题课录像上传到了 B 站，供其他班的同学和助教观看。

作业方面，我尝试引入了一些取材自计算机世界的题目，比如定义一组具有继承关系的类来表示抽象语法树的不同结点，还有些什么特别的题目我已经记不清了。在作业的测试方面，当时的 OJ 不支持太复杂的测试方式，唯一能做的就是在学生的代码的前面和后面增加一些东西，不过尽管如此我依然用 preprocessor/template metaprogramming 等手段实现了一些额外的检查，比如成员函数是否带有 `const` 、赋值运算符是否返回了 `T&` 等等，再比如重载 operator new/delete 来记录内存分配与释放从而检测内存泄漏。

这些改变当然是能看到效果的，但是很明显带来了很多问题，主要可以概括为一种**割裂感**，而且当时由于疫情封校封楼，我们助教之间和跟学生、跟老师的沟通都不是很顺畅，这使得这种割裂感变得更加明显。为什么割裂呢？因为这些改变毕竟始终是我作为一个助教单方面做的，是我单方面往课程里加入的内容。这门课究竟是什么内容，当然还是要由正课的课件和老师来定义，但是学生如果只上正课的话是不会学到我加的这些东西的，也就没法很好地理解作业里的一些额外的检测。并且，由于老师和其他大部分助教对这些东西也不是特别熟悉，这些代码规范性的检查就更像是我这个“C++语言律师”的自娱自乐，是我个人的追求而非被广泛接受的标准。学生如果拿着 OJ 上给出的这些额外的反馈去问自己班的助教或老师，会发现他们也给不出清晰的答复，渐渐地就不知道该怎么学了。

同时，我还发现有一些助教其实只是打算来水水，习题课可能就讲个半个小时，主要是混个助教经历和工资了事。当然，严格来说这没什么问题，确实是完成了本职工作的。

以上情况说明，想要在课程中教会学生写出符合标准的、规范的、可读和可维护的代码，就需要进行系统性的改革。

Spring 2022 还有一件非常重要的事，就是刘钰琦学长帮忙出了一道基于 OpenGL 的飞机大战游戏作为 hw8 和 hw9 ，他来搭好框架，学生负责实现游戏逻辑，不必去了解跟 OpenGL 相关的东西。刘钰琦学长也是一位非常负责任，很有想法也很有热情，并且有比较扎实的 C++ 基础的助教，在 Spring 2022 和 Spring 2023 中都和我充分合作做了很多工作。当时疫情原因取消考试，最终的给分完全由作业决定，其实给助教和学生都带来了很大的困扰，最后的 hw10 我们已经有点没活硬整了，而学生也被迫在考试周好不容易写完了飞机大战之后还要再写 hw10 。现在想来，当时如果能把飞机大战拆成 hw8-10 三次作业可能是更好的。这个飞机大战开启了 CS100 期末大作业的传统，其实挺合理的，不过这里面有挺多需要注意的问题，我们在后面几个学期里讨论。

### 查重

再说查重，这也是我非常纠结的一个问题。 Spring 2022 我们只在学期末对 hw4 和 hw8-10 进行了查重，并且只抓了那些相似度 80% 以上的，这个 bar 当然是非常高的，但同样是由于时间仓促，被我们判定为抄袭的会直接给那道题置 0 分而没有 argue 环节。这个零分影响是很大的，因为那个学期的最终给分里作业占 100% ，对相当多的同学来说作业做到 95+ 还是不难的，在这种情况下假如 hw8-10 里有哪一次零分就直接少了 12 分，几乎就在 B 以下了。但是假如是 hw4 的一道题置 0 分，那就只少了 $7/3$ 分，实际上还能在 A- 以上。总之这个查重的政策有非常大的问题，但 80% 这个 bar 实在是太高了，所以也没有引发太多的不满。

但我对于基础课里搞查重一直持有一些保留意见，特别是当时对 hw4 的第一题查重的时候我变得非常犹豫，因为那题考察了本不属于 CS100 的知识点——搜索。那题实际上来自于 NOIP2015 的【斗地主】，而出那道题的原本目的是为了考察递归。不得不说，这是一道出得非常失败的作业题，因为从“递归”这一具体的技术到“搜索”这种算法思想之间的鸿沟是 non-trivial 的，而题目里又没有给出任何引导，并且这道题本身写起来也比较麻烦，有很多零碎的情况要考虑。结果我们发现有好几份提交都抄了洛谷上的题解，有一些能看出来其实是自己尝试理解题解、对着题解边学边写的，但都被判了非常高的重复率。诚然，如果把 NOIP2015 的选手提交拿出来查一查重，是不可能有那么多 80% 以上的，但是 CS100 的学生们压根就没有学过搜索，除了学习那些题解之外还能有什么办法通过那道题呢？而且他们摸编程才摸了一个月，哪有什么自己的编程习惯呢，对着别人的代码写出来又怎么会有很大的区别呢？

回想一下自己学编程的过程吧。最初读《C++ Primer》的时候，我就是不断地对着那些示例代码学习，把它们遮住然后尝试自己写出来，再对照着看有什么区别，如此反复。后来打竞赛的时候，很多算法和数据结构都是先背下来一份模板代码，然后慢慢理解，渐渐地搞明白其中的每一个细节，并结合自己的编程习惯，最终内化为自己的知识。人类的学习本就是在模仿和借鉴的基础上的，把抽象的算法、思想以及语言特性转化为具体的代码是一种能力，是需要培养的。“你们可以交流思路，不能交流代码实现”，说得好听，但是对刚接触编程的同学们来说现实吗？

查重制度的存在，亮明了课程团队对于学术诚信的重视态度，防止了一部分在“抄”与“不抄”之间摇摆不定的学生滑向“抄”一边，惩罚了一部分又菜又懒指望不劳而获甚至恶意窃取他人成果的学生；但如果查重的规则过于严厉，也会妨碍一些好学生正常的借鉴和交流；同时，必然会有一些确实做不出来但懂得如何抄能绕过查重的学生成为漏网之鱼。一些助教在出题时甚至会把查重考虑在内，为了出一道让学生们不会写重的题而绞尽脑汁。因此，我们必须认真设计合理的查重制度，并将其与作业题目有机结合。比如，将作业题目按难度或类型进行分类，刻意出一些不查重的经典题、简单题、课件/习题课原题，让学生在这些题里练习那些最基础的东西。对于这些题，我们还可以有更丰富的评分手段，比如和助教进行“线下 check”——更正式的说法应该叫“代码面谈(code interview)”。对于中档题和难题，尤其是学生的发挥空间大、代码规模稍大的题目，本身抄袭不被发现的成本就会增大，我们在这些题目上进行查重也是合情合理的。这件事要在一开始就强调给学生：**简单题正是为了给你们锻炼基本功用的，如果你通过抄袭绕过了这个阶段的练习而没有掌握必要的技能，到中档题和难题你将会原形毕露**。后来进入了 AI 时代，以上这些对于管控学生对 AI 的依赖也是一样的，后面会提到。

此外，查重要早查、多查，处理方式要有梯度，不能像 Spring 2022 这样只在最后查一下。假如 hw4 、 hw7 、 hw8 有中档题和难题，那就在这三次作业刚结束的时候立刻查重，一种可能的处罚方式就是 hw4 只给被判重的那一道题记 0 分， hw7 给当次作业记 0 分， hw8 不仅要记 0 分还要倒扣；抄袭者和被抄袭者是否同等处罚，可以视情况而定。在执行的过程中要及时在 Piazza 上向全体学生公布查重和处罚情况，以示警告，但是注意保护隐私，不要公布名字；具体扣了多少分我建议也别公布，这样后续还可以调节。

## Spring 2023 ：改革的开始

### 基本情况，以及 C 的部分

这一学期的改变，首先从助教和习题课的模式开始。在组建助教团队的时候我就特别注意选择一些水平比较高、愿意在助教工作上花时间的人，把一些只是想来混助教经历的研究生拒掉了。习题课方面，我们将学生分为 7 个班，每个班 40 人左右，由 2-3 位助教负责，且其中应该包含一位高水平的助教，以这种“老带新”/“强带弱”的模式来平衡不同班级之间教学水平，也便于管理和落实一些改革措施。每周我们会开一次会，我把下周习题课的内容给所有助教先讲一遍，既讲知识点本身，也讲某些关键内容如何更好地教给学生。

我再一次重做了习题课的课件，工具从 $\LaTeX$ beamer 变成了 [Marp](https://marp.app/) ，并且这一次的主要参考资料从书籍变成了 cppreference.com 和标准文档。特别重要的是，我把习题课的语言由英文变成了中文，但是术语和引用的内容依然是英文的，这类似于[蒋炎岩老师](https://jyywiki.cn/)的操作系统课的模式。我的想法是，我们需要让学生渐渐熟悉全英文的专业内容，但是更重要的是先把知识点准确地教授给学生，不能本末倒置。不少学生在这个阶段用英语学习专业知识是非常费劲的，他们仍需要经过“英文 $\to$ 中文 $\to$ 概念”的转译过程。我们的正课的 slides 是英文的，习题课用中文，这样结合我认为是比较合理的。

作业方面，首先是 OJ 变了。上科大 ACM 社团从这时开始兴起，他们部署了 [Hydro OJ](https://github.com/hydro-dev/Hydro) ，我们的助教团队中正好有熟悉这一 OJ 的 ACM 成员，于是我们也部署了一个 Hydro 。从 Fall 2023 开始我们索性直接用他们 ACM 的 OJ 。 Hydro 尽管仍然是为算法竞赛设计的，但实际上已经具备足够的灵活性，可以在编译时和运行时执行任意的脚本，并且在编译时执行的脚本的输出会给学生看到。同时，我们尝试增大了作业的题量，尤其是前几次 C 的作业从原来的每次 3 题增加到每次 5 题，但是题目的难度下调，多出简单题，要避免 Spring 2022 时出的那些逻辑极其复杂的字符串处理题。我认为前期需要有足够的简单题的积累，在每一个简单题中练习一两个重要的新知识点，这种简单题可以简单到输入一列数之后倒着输出一遍、输出它们的总和和最小值，或者稍微难一点点的次小值，或者把这一列数当做一个多项式的各项系数再对输入的一列点分别求值等等。不要小看这样的简单题，一些上手慢的学生也需要花费大量的时间，同时在这些简单题上我们也更容易解释一些知识点和编程技巧。当时在 hw4 里我直接出了个实现标准库 `ctype.h` 里的那些函数（默认采用 ASCII 编码）和实现 `strcpy` 、 `strlen` 等函数的题，我认为这就是一个非常好的简单题，它既足够简单又能帮学生巩固对这些标准库函数的认识。如果要提高难度，可以增强综合性，也可以引入一些背景知识， hw4 的“内存泄漏检查器”和“Brainfuck 解释器”分别属于这两类，都是我认为很不错的题。内存泄漏检查器一题，通过 `#define` 把 `malloc`/`free` 等函数替换掉，借助 `__FILE__`/`__LINE__` 这些宏来记录内存的分配和释放，这涉及到宏和动态内存的知识，并且记录的信息是用 `struct` 存储的，这就把 `struct` 也带上了，同时还涉及到对一个数组的增删查，这实际上就是最最简单的数据结构思想。 Brainfuck 解释器一题则是在助教搭好的框架之下去模拟 Brainfuck 里的那几个操作，模拟的难度并不算太高，引入的有关图灵机的知识点起到了很好的拓展视野的作用。

hw2-3 的题目风格比较偏算法竞赛，具体表现在毫不相关的题目背景和比较死板的测试方式。当然作为比较基础的作业题它们是合格的，但我觉得还是可以改进。我理想中的题目背景要么来自现实世界，要么来自计算机科学，前者可以提升趣味性并增加学生对于程序和外界世界的联系的理解，后者可以开阔学生的眼界，而且有些知识确实只有在计算机科学的世界里才会得到更充分的应用。测试方面，这一学期我基本没有搞太多的变化，但是 hw3 的某道题和 hw4 的 Brainfuck 、 内存泄漏检查器这些题目我们下发了测试数据，学生可以自己测试。我觉得这是一个可以尝试的事情，学编程跟打竞赛还是有本质的不同，更何况竞赛训练平台（比如洛谷）也会开放题目测试数据的下载。要思考，究竟何种反馈方式能让学生更好地学习。

C 的部分就这样按部就班地过去了，期间我们也听到学生在 hw3 前后抱怨作业量大，这一点我到 Spring 2024 的部分来详细讲讲。学生也感觉到正课的 slides 质量远不如习题课，但因为 C 的知识点本身不复杂，二者结合着也就凑合学下来了。

### C++ 的部分

#### 课程内容的改进

C++ 部分的问题就很大了，这里我们首先要谈一谈对 C++ 的基本认识。众所周知 C++ 的第一个 ISO 标准是 C++98 ，第二个并且也是所谓“modern C++”的开端是 C++11 ，自 C++11 起三年更新一个版本， C++14 和 C++17 实际上都是在 C++11 基础上的修订，可以说 C++17 就是“C++11 想象中的自己”。 C++20 被认为是第三个“大版本”，其中加入了 coroutines 、 concepts 、 ranges 、 modules 等非常多的新特性，但与此同时 GCC 和 Clang 直到现在也没有完全支持所有特性，尤其是 modules 。总之考虑到编译器的支持也好，学生的学习成本也好，我们最终选择 C++17 作为这门课的标准。 Scott Meyers 在《Effective C++》的第一条款就说：“视 C++ 为一个语言联邦。”他将 C++ （当时是 C++98）视为四门语言的联邦，分别是 C 、 OOP 、模板、 STL 。到 C++20/26 恐怕还得增加几个“语言”，但好在这个分类对于 C++17 依然是成立的。许多学校，包括我们的旧课件，教的都是 C + OOP ，结果是什么呢？在 Spring 2022 ，到最后一次作业的时候我们看到有同学居然用 `for` 循环逐元素拷贝 `std::vector` ，我们还看到有 21 级 CS 的同学在 CS101 上写代码时动辄定义 class + 拷贝控制成员却不知道可以用标准库容器来代替 `new`/`delete` 。必须要指出，“C++ 是一门面向对象编程语言”这句话应该理解为“C++ 支持 OOP ，并且 OOP 在 C++ 中被广泛应用”，但若要给 C++ 从编程范式的角度下一个定义，它只能是 **multi-paradigm language** 而非 object-oriented language 。 Bjarne Stroustrup 在“The D&E Book”（《The Design and Evolution of C++》）中对 C++ 给出了这样的定义（4.1 节）：

> ... the fundamental aims of C++:
> 
> - C++ makes programming more enjoyable for serious programmers.
> - C++ is a general purpose programming language that
>   - is a better C
>   - supports data abstraction
>   - supports object-oriented programming

class $\neq$ OOP 。[孟岩老师的这篇文章](https://blog.csdn.net/myan/article/details/5928531?spm=1001.2014.3001.5506)说得很好（上古时期的 CSDN 还是有如此高质量的内容的），我们这里主要看看他对 OOP （即“对象范式”）的解释：

> 对象范式的两个基本观念：
>
> - 程序是由**对象**组成的；
> - 对象之间互相**发送消息**，协作完成任务；
>
> 请注意，这两个观念与后来我们熟知的面向对象三要素“封装、继承、多态”根本不在一个层面上，倒是与再后来的“组件、接口”神合。

class 本身是一种实现抽象（“data abstraction”）的工具，它当然是定义对象从而实现对象范式的有效且主流的工具，但并不是用了 class 就是对象范式。 C++ 的 class 有很多种用法，其中有相当多的一部分跟 OOP 没有什么关系。

那么 C++ 的课程应该教些什么、按什么顺序教呢？我认为《C++ Primer》（注意没有“Plus”）（第 5 版）的章节安排是非常有借鉴意义的。《C++ Primer》在第 3 章就讲了 `std::vector` 、 `std::string` 和基于范围的 `for` 语句，甚至放在了数组之前；接着第 4-6 章讲了基本的表达式、控制流语句和函数；第 7 章讲了类的基础；第 8 章是 IOStream 库；第 9-11 章是一般意义上的“STL”，即顺序容器、算法、关联容器，其中第 10 章讲算法的部分讲了 lambda 表达式；第 12 章才讲动态内存管理，并且是把智能指针放在 `new`/`delete` 之前的；第 13 章讲了拷贝控制，也包括移动操作；第 14 章讲了重载运算符；到第 15 章，标题是“OOP”，这时才把继承、多态等特性端出来；第 16 章是模板；后面则是一些零散的特性和库设施。我们会发现，《C++ Primer》的内容组织非常明确地传达了一个信息，那就是**库设施优先于语言细节，复用库代码优于自己动手**。《C++ Primer》还把现代 C++ 的新特性充分地融合到原有的知识中去，而不是放在末尾作为什么“高级主题”。此外，这本书有很多对于语言设计以及代码质量和最佳实践的讨论，它不仅回答了 C++ “是什么”，还解释了它“为什么是这样”。我并不是赞同对《C++ Primer》的内容安排直接完全照搬，但是其中体现的思想是需要重视的。

还需要考虑到容量的问题。讲完 C 之后还剩下 10 周时间，对 C++ 的内容必然要有所取舍，我们要选取最重要的、最想让学生熟练掌握的内容优先讲，并且在每一个部分里也要有一定的取舍。比如 STL 当然是很重要的，但是我们没法把 STL 里的全部细节都讲清楚，比方说 `std::map` 和 `std::unordered_map` 背后的原理等等。学生的知识背景也是要考虑到的因素之一——在上这门课的时候他们还没有学过算法和数据结构，甚至不知道什么是链表。在旧的课件中链表也是 CS100 的一部分，但我把它移除了——这种综合了 `struct` 、指针和动态内存又带一点数据结构思想的东西，放在前六周喂给刚搞清楚 `*` 和 `&` 的区别的初学者实在是不太合适。我特别想问问往届在这个时候学了链表的学生，有谁能说明白我们什么时候需要使用链表？

综合以上这些考虑，我大致定了这样的顺序：

1. 首先完成从 C 到 C++ 的过渡，包括
   - C++ 中的 “C” ，如函数、类型等等与 C 的区别
   - `std::vector` 、 `std::string` 、 基于范围的 `for` 语句
   - `new` 和 `delete` 。我没有把它放到智能指针后面，一是因为我们实际上已经在 C 中接触过了动态内存，可以现在就拿出来与 `malloc`/`free` 进行对比；二是需要为接下来讲类的基础时的 RAII 部分做铺垫。
2. 接下来讲类的基础，重点放在**数据抽象**上，包括
   - 数据成员、成员函数、构造函数、析构函数
   - 拷贝操作
   - static 成员、类型成员、 friend 等零散特性
   - 右值引用和移动。这里不紧跟在拷贝操作后面讲，主要是为了用一些零散的特性起到一个缓冲的作用，免得学生一口气学拷贝 + 移动可能会吃不消。
3. 智能指针，这正好作为刚刚讲完的拷贝和移动操作的一个重要实例。但是我们不讲 `std::weak_ptr` 。
4. 重载运算符
5. 迭代器和算法，作为 STL 的开头。但是我们并不讲得太深，只讲一些常用的部分，当然也包括如何向算法传递定制化的操作，包括 lambda ，但是只讲最简单的用法。
6. 继承和多态，这里就是真正讲 OOP 的部分了，重点当然是虚函数动态绑定，但同样重要的是理解其背后的思想，何时使用继承、何时使用组合。这里我们特别拿出《Effective C++》的条款 32 （“public 继承建模 ‘is-a’ 关系”）和条款 34 （“区分接口继承和实现继承”），在课上讲解这两个条款并让学生课后精读。我们不讲多继承和虚继承。
7. 进一步认识 STL ，包括顺序容器和关联容器
8. 模板
9. 标准库中的其它设施
10. 编译时计算和元编程

注意，虽然是十周讲这些内容，但并不是每个部分一周。我们的考试内容是到第 6 部分“继承和多态”，在这以后的内容就更多地是起到开阔眼界的作用，供一部分比较感兴趣的同学们学习，并且可能会在编程作业中涉及，但不需要深入地掌握了。实际上我还有一份关于异常处理和异常安全性的课件，目前是放在了第 7 部分和第 8 部分之间，但我觉得跟前后内容有些脱节，也不知道怎么处理更好。

在从 C 过渡到 C++ 的时候，还需要处理好 C 和 C++ 的区别的问题。 C++ 中的“C”和 C **并不是完全一样的**，不能指望一个 C 程序能用 C++ 编译器编译成功。然而二者之间并非存在着不可逾越的鸿沟， C++ 中的 C 实际上包含了对 C 的改进与妥协。拿语言中最重要的部分之一——类型系统来说， C++ 明确反对指针之间的隐式转换，但同时对于整数之间的窄化的隐式转换采取容忍的态度。在《The Design and Evolution of C++》第 2.6.1 节中说到，他当时曾试图禁止一切隐式的窄化转换，但是

> The experiment failed miserably. Every C program I looked at contained large numbers of assignments of `int`s to `char` variables. Naturally, since these were working programs, most of these assignments were perfectly safe. That is, either the value was small enough to become truncated, or the truncation was expected or at least harmless in that particular context. There was no willingness in the C with Classes community to make such a break from C. I'm still looking for ways to compensate for these problems ($\S 14.3.5.2$).

最后提到的第 14.3.5.2 节，正是他围绕 `static_cast` 展开的讨论。最终的方案是大家都知道的：隐式的窄化转换在一般的语境中通常会产生 warning ，而在 C++11 所引入的大括号初始化语境中是 error 。这个例子很好地展现了 C++ 对于 C 的特性的扬弃哲学。在讲到 C++ 中的“C”和 C 的区别时，要从这样的角度去解释这种区别产生的原因，而不是死记硬背。同时，这些内容其实和我们始终强调的 best practices 是有机统一的： C 中的很多所谓 best practices 到了 C++ 里就成了标准强制的行为。此外，**不要在考试和 quiz 里考察 C++ 与 C 的某些微妙的区别**。在那些有区别的地方， C++ 的规则几乎总是更好，我们直接在 C++ 的环境下拿出好的行为来考察学生对其的理解即可。

在 Spring 2023 ，刚进入 C++ 部分的时候我只是想着要在原有的课件基础上进行一些改进，但改了一两份之后就发现已经完全看不下去，遂决定自己动手全部重做。当时是非常仓促的，没有足够的时间精雕细琢，只是先把内容都做出来，至于更好的排版、课时容量的裁切，那都是后来的事。那个时候老师在正课上也就是“讲到哪算哪”，当然其实这么做的课程并不少见，这也不是什么大问题。尽管不完善，学生还是给出了相当正面的反馈，他们觉得 C++ 部分的课件比 C 的部分清晰多了。我在做课件的时候特别注意了这一点，对很多概念性的东西给出了像数学讲义里那样的正式陈述，比如（下面这个例子来自我后来在 Fall 2023 重做的 C 部分的课件）：

> Let `p` be a pointer of type `T *` and let `i` be an integer.
>
> - `p + i` returns the address equal to the value of `(char *)p + i * sizeof(T)`. In other words, pointer arithmetic uses the unit of the pointed-to type.
> - If we let `p = &a[0]` (where `a` is an array of type `T [N]`), then
>   - `p + i` is equivalent to `&a[i]`, and
>   - `*(p + i)` is equivalent to `a[i]`.

再比如一些语言特性的文法：

> Syntax: `ReturnType FunctionName(Parameters) { FunctionBody }`

像这样的内容，大一的学生凭借基本的数学素养是完全能够看懂的，我们也应该用这种方式鼓励学生去适应这种形式化的表述，无形中培养他们自己去阅读标准文档的能力。当然，足量的示例、图片等作为辅助也是必要的，课件基本采用了类似于《C++ Primer》和“Effective”系列的语言风格，通过对语言特性的各种讨论来加深学生的理解，同时也告诉学生我们在分析程序片段和语言特性的时候应该从哪些方面思考。

你可能在学术汇报相关的课程上听说过，把过多的文字放在 slides 上不是一个好主意。但是课件和汇报展示不同，课件需要供学生反复地看、做笔记。我们有些课程的课件（这里特别批评 Fall 2023 CS282）带有强烈的防自学属性——一门偏数学的课程，课件上连记号定义都没有，什么都不解释就摆一串公式，重要的定理和定义也不给出一个规范正式的陈述，假如上课走神了或者没来，光看那个课件根本就学不会。

正课课件是这样，习题课的课件依然是起到复习和补充的作用。特别要注意，有个别重要内容，比如**值类别**，我是放在习题课上正式讲的，是在紧跟着 Lecture 12 的那节习题课，这一点一直延续到了 Spring 2024 。后续助教在准备习题课时，一定不要遗漏这些内容，可以去正课课件里搜索 `recitation` 看看有哪些内容说了要在习题课上讲。习题课和正课不同，由于每个学期的具体进度都会有区别，受到作业的安排、节假日等多重因素的影响，**我们没法做一套对每个学期都适用的习题课课件**。习题课课件可以根据学生的需要讲评某些作业题，现场演示某些例子，或者加入一些有关工具使用、调试技巧、多文件编译等额外内容的讨论。

**习题课不能只有补充而没有复习**。复习不是只有在临近考试的时候进行，而是在学习的过程中不断地复习。 Spring 2022 的时候有些助教之所以习题课讲半个小时就下班，就是因为他觉得有些内容学生都在正课上学过了，就不用再讲一遍了。然而我始终记得我以前打算法竞赛时，有些知识点我反复学了三四遍才学会——包括数论的入门、后缀数组、傅里叶变换，每次我都感觉自己好像看懂了，但实际做题的时候感觉那些知识点离我还是非常遥远。学习一个知识点是需要反复的，听不同的老师讲，从不同的角度学习，尝试与不同的知识点进行联系，体会它在不同的应用中扮演的角色，正如《实践论》中所说：“只有感觉的材料十分丰富（不是零碎不全）和合于实际（不是错觉），才能根据这样的材料造出正确的概念和伦理来。”因此，习题课上把某些关键的知识点重新讲一遍，我认为是十分重要的，而且这并不是对正课的简单重复。

#### 作业设计

C++ 部分有一个重要的例子，就是所谓的 `Dynarray` 类，这个名字其实是 dynamic array 的简写，来自于 C++14 的废案 [N3662 (`std::dynarray`)](https://www.open-std.org/jtc1/sc22/wg21/docs/papers/2013/n3662.html) 。不过 `std::dynarray` 实际上和我们的 `Dynarray` 不完全一样，它的提出者希望 `std::dynarray` 的实现可以根据情况选择使用栈内存作为一种优化。当时确实还有一个提案 [N3497](https://www.open-std.org/jtc1/sc22/wg21/docs/papers/2013/n3497.html) 试图把 C 的 variable-length array (VLA) 引入 C++ ，但由于它和 C++ 的静态类型系统实在是无法相容，最终没有被接收。扯远了，我们的 `Dynarray` 实际上有点像 `std::valarray` 但没有那些数值运算，又像不能扩容的 `std::vector` ，或者说是对 `std::unique_ptr<T[]>` 的封装，总之就是把 `new[]` 来的“动态数组”做了个封装，算是资源管理类最入门的练习。这个例子我们在正课的课件里有，习题课课件里也会复习一遍，作业里会连续使用，每一次在前一次的基础上增加一些新东西，并且考试里也可以继续用它作为素材。大约在 hw6 ， `Dynarray` 类应该包含所有基础的成员 + 拷贝控制成员，包括移动，这一次作业我们采用 OJ 测试 + 代码面谈，学生和 TA 一对一就代码中的细节进行问答，包括 `Dynarray` 类的 class invariant 、辨认出某些特殊成员函数并解释其行为、解释某些地方为什么加 `const` 等等、解释某些特殊的特性的使用（如委托构造函数、 `std::distance` 、 `new int[n]{}` 的花括号等），以确保学生确实理解了其中的细节而不是直接从课件抄的或者 AI 生成的。要不要把它改成像 `std::vector` 那样可以扩容的呢？仔细设计一下也许可以，但我当时没有这么做，一方面是希望它保持简单，另一方面要实现这种扩容实际上涉及到区分内存分配和对象构造，正确的实现应该让 `size()` 到 `capacity()` 这一范围内的部分处在未初始化的状态，这就不能直接用 `new[]` 了，也不应该用 C 的 `malloc` ，而是应该用 `std::allocator` 相关的东西，较真一下的话还是有点复杂的。当然，毕竟在讲到模板之前我们的 `Dynarray` 存放的都是 `int` ，我们也可以不考虑这些事情而直接 `new[]` ，这样一来其实也不会增加太多的复杂性，不过最好有一份辅助性的阅读材料把它和 `std::vector` 的区别告诉学生。反正一直到 Spring 2024 我都没有在 `Dynarray` 里支持扩容，后面的助教可以考虑一下。

`Dynarray` 中还有一个我一直很纠结的点，就是拷贝赋值运算符的异常安全性。初学者的实现通常是先 `delete[]` 旧资源，再分配新内存并拷贝数据，这显然不满足**自我赋值安全性**。要解决这个问题，最简单的办法是加上 `if (this != &that)` 的判断，但这依然不够好——如果 `new` 抛出异常，对象状态可能无法恢复，这就缺乏**异常安全性**。实际上，只要调整顺序，先复制数据再删除旧数据，就能同时满足自我赋值安全性和强异常安全保证；而如果使用 copy-and-swap idiom ，甚至可以用一个函数同时实现拷贝赋值和移动赋值运算符。然而，异常处理并不在我们的课件中——我始终觉得异常处理是有价值的内容，但不知道该放在哪里。在 Spring 2023 ，我的解决方案是在作业中增加一个附加题，专门测试这个运算符的异常安全性，通过的话可以加 10 分（在总成绩中占 0.007 ，影响微乎其微）。很多学生会把自己的满分程序交到这个附加题里试一试，而我们的目的就是通过测试失败来告诉他们："你的实现还不够好。"至于他们是否愿意去查阅资料、改进实现，那就由他们自己决定。为什么不让这个附加题占 0 分呢？因为那些认真学习异常安全性的学生值得一点点认可和鼓励。

在 Spring 2023 的 C++ 作业中，我还给每次作业都出了一份客观题，大约 20 道选择，都是不定项。这些客观题都挺难的，需要对相关的知识点有非常扎实的理解。我觉得这些客观题出得不是太好。 Fall 2023 我还是把这些题拿出来凑了个数，但到 Spring 2024 我就去掉了，而是改成在 C 的部分出客观题，让这种作业形式服务于更加基础的知识点。 C++ 部分还可以考虑出这样的客观题，但应该出得更简单一些，不要抠得那么细，而且不定项选择这种无比恶心的题型实在不应该保留——一道 4 个选项的不定项就如同 4 个判断题，而且交上去你也不知道到底是哪个选项判断错了。稍后讲到考试的时候我们还要再谈到这一点。

Spring 2023 的 C++ 作业里有几个题出得挺认真，比如赵清澄的“生命游戏”。遗憾的是这个题的继承体系里没有数据成员，它这个继承体系更像是体现了某种设计模式（具体是啥我忘了，我没有保存这道题的材料），而它实际上完全可以用 `std::function` + lambda 等等以函数式的风格简单地实现，那种用继承的实现方式是相对传统的。 hw7 出了一道“表达式树”，这个例子有必要好好讲讲，我在 Spring 2022 就出了这样一个题，但当时没有讲智能指针，对于几个结点类的拷贝控制也没法做得很好，而到了 Spring 2022 就万事俱备了。这个表达式树的例子，在《Ruminations on C++》这本非常老的书的第 8 章就有，第 9-10 章则是同样的方法在另一个问题中的应用，《C++ Primer》第 15 章的支持与或非的文本查询实际上也是这回事。《Ruminations on C++》写作时间非常早，甚至早于 C++98 ，所以那本书里他们是手动实现了引用计数从而完成拷贝控制的，而《C++ Primer》则使用了 `std::shared_ptr` 来自动管理资源，我们当然也是采用后者。在 Spring 2024 ，我在这道题的基础上又出了一道更偏编译原理、结合了 Visitor Pattern 的题作为延续，那题处理的是程序中的表达式而非数学表达式，需要在我的引导下学一点点新知识，但其实要写的部分很简单。我认为在 hw7 这个时机，放两道像这样的题是非常合适的，它不是“Shape + Circle + Rectangle”那种特别弱智的，正好可以跟有着更大规模继承体系的 hw8 进行衔接。以后的助教可以继续沿用这两个题，也可以去翻翻我上面说的两本书里的另外两个例子，看看能不能出成题。

hw8 依然是刘钰琦出的 OpenGL 大作业，这一次出的是植物大战僵尸。这里涉及到一个重要的问题，就是多文件编译和 CMake ，这个内容在旧的课件上是有的，所以 Spring 2022 的时候我们就顺理成章地出了，但是新的课件更注重语言本身，就像《C++ Primer》那样，对语言之外的东西讨论很少。 Spring 2023 的第 11 次习题课简单介绍了一下多文件的原理，演示了一个最简单的例子，不过在这个阶段我们没法把动态库静态库之类的原理解释得很清楚。并且，我自己是在 Linux 上演示的，但最好能有一份 Windows 的演示，这样学生更好复现。至于 CMake ，我们的方案是找几个助教录了一些视频，作为 hw8 的辅助材料给学生看，同时我们助教团队尽量保证帮每个来问编译环境的学生解决编译环境问题。当时，我们发现 hw8 的代码在 Windows 上用 MinGW 无法正确链接，于是花了很大的工夫教学生搭建 VS 的环境，以及满足一些学生用 VSCode 写、用 VS 编译的需要。然而后来我们发现这个问题其实是可以修复的，这一点我们在 Spring 2024 细说。

hw8 实际上还有一题，是有关模板的一些客观题。关于模板的作业我想了很久，包括是否应该把 `Dynarray` 变成一个模板类 `Dynarray<T>` 让学生来写，但考虑到时间限制，我还是认为应该把重点放在植物大战僵尸上，所以最终决定把模板版本的 `Dynarray` 代码实现发给学生，并让学生回答一些与此相关的客观题，也就是说学生只要能看懂这份代码，不需要自己动手写。

植物大战僵尸的评分，刘钰琦设计了一套 OJ 测试 + 代码面谈的模式。首先，学生需要在 OJ 上提交代码进行功能性的测试，这里刘钰琦做了一套相当复杂的测试，把游戏的底层渲染交互框架换成了一套接口相同的测试框架，并且在很多概率性的功能上（如不同僵尸产生的概率）都花了很多心思进行测试。然后，学生需要在 OJ 上填写一份问卷（采用 Hydro 客观题的模式），包含自己是否遵守了各类非功能性要求，其中涉及文件组织、代码风格和结构、最佳实践的遵守等等。这份问卷相当于是辅助代码面谈的材料，在代码面谈环节助教可以快速核实这些非功能性要求是否被满足，另外还可以就代码中的一些细节与学生进行讨论。

这套评分模式在当时看来挺好的，但它隐含着一些问题：
1. 测试要求学生完全不能修改与渲染交互框架相关的任何东西，连这些文件的组织方式都不能改，但实际上这里面是有发挥空间的。
2. 测试实际上是通过在测试框架里开了一些后门直接读取程序中的对象的值来完成的，而不是看游戏表现得正常不正常。学生有时会感觉自己的游戏玩起来挺正常的，但不知为何就是不能通过测试。这非常打击学生的自信心，而我们本意是希望学生从这个游戏实现中感受到一点成就感和乐趣的。
3. 测试标准是确定的，这就要求学生必须严格按照出题人的要求实现这个游戏，不能改动其中的功能或数值，这缺少一些灵活性。
4. 在代码面谈环节，助教做的大部分工作仍然是机械性的，比如检查学生是不是充分且正确地使用了智能指针、是否避免了 magic numbers 、是否在某些功能上进行了合适的抽象而避免大段重复等等。

有没有一种可能，其实功能的正确性应该让助教在代码面谈环节玩一玩来检查，而那些代码风格、文件组织之类的情况反而应该自动化地检查？

实际上整个学期的作业中，尽管我们到处强调代码风格、最佳实践，但除了一些简单的基于元编程的检查和代码面谈，我们没有任何办法做更加细致的检查。我们需要真正的工具。

### quiz 和考试

这一学期的时间安排实在是比较混乱，特别是在刚进入 C++ 时临时起意重做课件，且课件在做的时候也没有把课时安排计划得很清楚，导致“期中考试”拖到了 15 周。更加糟糕的是，我出了那些难度颇高且都是不定项选择的客观题之后，直接把这种题目风格带进了考试，对于作业和考试的区别、题目的风格和形式等等都缺乏深入的思考。这份试卷 80% 的题目都是我出的，阅读量极大，综合性极强，需要对知识点非常熟悉才能反应得过来。如果用“脑中编译”概括我出的题目，那剩下 20% 刘钰琦出的就是“脑中运行”，这就更加雪上加霜了，尤其是有一道手写代码的题我们还低估了正解的代码量。此外，我们当时还十分保守地采取了半开卷的策略，只允许学生带两张 A4 cheat-sheet ，这实际上也是压根没有动脑就沿用了往年的政策。于是，这场考试就这样变成了一场“C/C++ 语言律师大赛”。

不过，尽管有着诸多问题，考试里有一些题目我觉得还是不错的，值得说一说。比如选择题的第 3 题，我们定义了一个 `student` 结构体，现在想要为它设计一组 `student_new` 和 `student_delete` 函数，分别表示在堆上创建一个 `student` 对象和销毁一个 `student` 对象并释放其内存。四个选项给出了四组 `student_new` 和 `student_delete` ，问其中哪些是正确的。这需要学生将两个函数联系起来思考，要理解为什么 `student_new` 里的改动会引起 `student_delete` 里的改动。第 4 题直接给了一个在程序中出现多次的模式（不断移动指针，直至某条件满足），问我们怎样可以提取出一个函数，以及如何使用这个函数。四个选项分别给出了四种函数实现以及它们对应的使用方式。这两道题都不是简单的编译判断，还考察了对于程序中多个部分之间的联系的认识，并且它们的场景也都是非常典型的，是在实际的程序中确实经常出现的。再比如最后一道大题，我引入了 ref-qualified member function 这一课上没有讲过的语言特性，给出了其语法定义和示例，然后考察在 `Dynarray` 上的一个简单的应用。这道题既考察了学生对于左右值和成员函数的理解，又考察了阅读文档定义的能力，还联系了大家非常熟悉的 `Dynarray` ，我觉得是一道不错的题。但遗憾的是，这张卷子的题量实在是太大了，特别是阅读量很大，相当多的学生没有足够的时间做这道题，尽管它其实不是很难。

这学期只进行了一次 quiz ，是在 C 快结束的时候，题目的风格跟上面所说的考试和客观题也比较类似，当然难度和综合性都比较低，毕竟那个时候学生还没学太多的东西。

考试、 quiz 的情况和学生对客观题作业的反馈，都表明我们在这些环节需要大改。

### 给分

老师非常信任我，把最终给分的工作也交给我来做了。我的给分方式是算出加权总和（其中，习题课签到的 2% 我全都给了），从高到低排序，求出一阶差分，在一阶差分的局部极大值处考虑划线。这样可以避免诸如“A 的最后一名和 A- 的第一名只差 0.01”这样的情况。 20 分是及格线，也就是说认真完成三次作业就可以及格，这个标准相当低了。

各等第的人数如下：

| A+  | A   | A-  | B+  | B   | B-  | C+  | C   | C-  | F   |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| 21  | 105 | 17  | 33  | 29  | 17  | 22  | 12  | 14  | 14? |

最后 F 的人中，有一些我不确定是真 F 还是退课。当时老师让我 A- 及以上不要给超过 50% 。

可以看到， A/A+ 率高达 43% 。这是我的仁慈吗？是我“自己淋过雨，所以懂得为别人撑伞”吗？**不是**。我并不是什么狂撒 4.0 的“绩点侠”。我之所以给如此多的 4.0 ，是因为这门课的作业和考试确实在广度和深度上都很强，能按部就班跟下来的学生我认为都值得 4.0 。按照作业满分 + 考试平均算出来的成绩正好在 A 的最后十几名的位置，这正合我意。

## Fall 2023 ：最终方案的雏形，以及秋学期的特殊性

从 Fall 2023 开始秋学期也开设 CS100 。秋学期和春学期在课程大纲上没有什么差异，但是由于选课的学生不同，我们的教学方式和目标当然是要有所区别的。秋学期选课的学生以 EE 、 BME 等其它专业为主， CS 的学生是不会上秋学期的 CS100 的，除非有特殊情况。

CS 和非 CS 学生的区别是什么？非 CS 的学生中当然有一部分是编程天赋稍差的，但是 CS 的学生里就没有吗？恐怕有不少 CS 学生其实也只是跟风选了这个专业吧？而且 EE 和 BME 这些专业本身也并非一点代码都不写，这些专业中的很多知识和技能跟编程所需要的思维方式也是相通的。至少在 CS100 这样的编程基础课上，我不认为 CS 专业和 EE 、 BME 等专业的学生有什么根本上的隔阂。至于后来一些创意与艺术专业的学生也被要求学 CS100 ，这我确实也持怀疑态度，不过我也没有很清晰的了解。

但是， CS 和非 CS 学生在**学习目标**和**时间安排**上是有区别的。 CS 学生大概率会需要频繁地与代码打交道，无论是自己写还是阅读、测试。哪怕有些人未来用 Python 远多于 C/C++ ，学好这门课对于培养对编程的理解和好的编程习惯也是有重要的意义的。对于 CS100 这样一门基础的 C/C++ 编程课， CS 学生有充分、正当的理由投入大量的时间和精力来学习。对非 CS 专业的学生则不然，他们的学习重点可能是数电、模电或者什么其它的内容，他们不同的人之间的学习目标的差异是相对较大的，其中有一些其实也需要打好扎实的编程基础，而有些人可能大致了解一下编程是怎么回事就行了。

因此，秋学期我们的策略是：**内容不变，但要求降低**。课程内容是和春学期一样的，不能减少，因为仍有一部分学生需要或愿意建立扎实的 C/C++ 编程基础。要求降低，也就是作业和考试的难度降低，以减轻他们的负担。同时，最终的给分也要适当调整。 Fall 2023 最终的 A/A+ 率为 35% 。

这里我们还有必要驳斥一下一些比较极端的功利主义言论。之所以要在这里专门讨论这个问题，是因为它关系到我们如何理解基础课的价值，也关系到学生在面对困难时能否坚持下去。一些人认为“我的研究方向主要是用 Python 的，不需要学 C/C++”，或者“我的专业压根就不需要学编程”。编程是这个时代最重要的技能之一，这毋庸置疑；而编程中体现的**抽象思维能力、系统化与分解能力、精确与严谨的逻辑思维能力**，更是这个时代最重要的能力之一，因为当今世界各个领域都在信息化、智能化。哪怕以后很多编程任务会由 AI 来完成，培养这些思维能力也是大有裨益的。

那么，为什么要学 C/C++ 而不是其它语言呢？C 语言作为一门抽象粒度较低的语言，它会告诉你最基本的有关计算机系统的知识——内存是怎么回事、数据是如何表示的、指针意味着什么。而 C++ 会向你展示那些看似高级的语言特性是怎么在 C 这么低级的语言上创造出来的，抽象为什么是我们解决问题的强大工具。比如，你在 Python 中使用的 `list` 、 `dict` 等数据结构，在 C++ 中你会看到它们是如何通过 class、模板、内存管理等机制一步步构建出来的；你在高级语言中习以为常的自动内存管理，在 C++ 中你会理解它背后的 RAII 思想和资源管理的权衡。**这种理解会让你在使用任何语言时都更加清醒，知道自己在做什么、为什么这样做。**因此，哪怕你认为自己大概率不需要用 C/C++，以 C/C++ 为培养编程基础的第一门语言也是很不错的选择。当然，我并没有说这是最好的选择。

此外，类似的问题也在很多其它基础课里被问到：为什么要学数学分析？为什么要学线性代数？蒋炎岩老师给了[一个不错的回答](https://jyywiki.cn/OS/2023/build/lect1.ipynb.html)：**重走“从无到有”的发现历程**。这句话的意思是，基础课的价值不仅在于教会你某些具体的技能，更在于让你理解这些知识是**如何被创造出来的**、**为什么需要它们**。对于 C/C++ 也是一样：学习 C 让你理解计算机是如何表示和处理数据的，学习 C++ 让你理解抽象是如何一层层建立起来的。这种“从无到有”的过程，培养的是一种**探索和创造的能力**，而不仅仅是编程本身。

### 课件更新

Fall 2023 的第一个主要改变是课件更新。我把 C 的课件也完全重做了一遍， C++ 的部分虽然没有特别大的改动，但也修订了一遍。这一次我仔细裁切了每一份课件的长度，让它在 40-50 页左右，每次上课正好讲一份课件。每份课件都有明确的 summary ，部分课件有 exercise 。我对照 C17 和 C++17 标准认真地检查了课件内容。我将所有“最佳实践”都显式地标上了“[Best practice]”。一些特殊的说明，比如对标准内容的简化或者涉及到标准的变动的，用红色角标标注并在课件的末尾给出注解。

我在课件里声称我们的课件内容全都符合 ISO C17/C++17 标准，但其实这是带来了一些麻烦的。比如， ISO 标准没有规定字符必须采用 ASCII 编码，甚至没有规定一个字节一定是 8 位。但是我们如果在讲课的时候处处都强调“这个 `'A'` 不一定是 65”、“`char` 不一定是 8 位”，那就有点逆天了。诸如这样的内容，我们必须要把最常见的那一套实现作为“默认”的事实，而将标准的实际说法作为尾注，这样既不影响正常的教学，也给了少部分想要一探究竟的学生一定的指引，同时也保证了课件对于语言标准的遵守。幸运的是，这样的内容也并不是很多。

肯定有人会问，这个语言标准就非要遵守不可吗？我们又不是在培养语言律师。就直接在课件里说 `int` 是 32 位又能怎么样呢？首先，我还真见过 16 位的 `int` ，它实际上并不是非常罕见，尤其是如果有些学生接触了某些特殊的嵌入式系统，很多不常见的事情都有可能发生了，比如不用 IEEE754-binary 表示的浮点数。拥有一个严谨的标准对一门课来说是很重要的，这可以减少很多的麻烦。比如，数组越界访问到底会发生什么？按照语言标准，这就是 undefined behavior ，**句号**。从实践的角度，我们可以展开说说它在实际程序中的*可能*的表现，以加深学生对这种错误行为的认识。但假如要在考试题里让学生回答，那么答案就是“undefined behavior”，到此为止。如果没有这样一个明确的标准，那这题就无法回答了，因为在编译器的层面上，它可能会把这个越界的行为“优化”成任何别的行为；在操作系统的层面上， Windows 上和 Linux 上结果也未必相同……到底什么答案是对的？

有没有比语言标准更适合作为课程标准的，我不清楚，但是实践表明语言标准与大众普遍认识的 C/C++ 距离也不是非常遥远，我自己一边做课件一边把我自以为学过的东西重新按标准学习了一遍，也没有见到什么颠覆我认知的。我们已经成功做出了一套遵循语言标准的课程，因此如果你没有特别坚定的理由换别的标准，那就这样吧。我们的课程跟语言标准接轨，也就跟 cppreference 、 Stack Overflow 、 CppCon 等这个世界上最权威的 C++ 专家出没的平台接轨了，挺好的。

在 C++ 的部分，我在模板的前面插入了一份异常处理与异常安全的课件。这份课件是用 LaTeX beamer 写的，一方面它相比于 Marp 可以让我更好地演示 stack unwinding ，另一方面我依然觉得这个内容放在这比较突兀。到 Spring 2024 也依然是这样，我不知道究竟怎么安排比较好，也许直接把它改改，改成一份便于阅读的教程作为课程的辅助性内容给有兴趣的学生（包括那些想理解 Dynarray 附加题的学生）看吧。

有些课件的文件夹下是有 `notes.txt` 的，那是我做课件的同时写的一些注意事项，可能涉及到某些内容为什么这样写、我认为应该怎样讲等等。但实际上还有很多内容我在设计的时候思考了很久，却没有来得及把我的想法记录下来。如果有什么疑问欢迎在我的 GitHub 的 repository 里提 issue 或者用其它任何方式联系我。**我产出的内容几乎没有什么是未经过深思熟虑的**，非常欢迎有想法的人来和我探讨。

课件在这一学期完成了更新之后，基本就没有什么大的改动了，一直沿用到了 Spring 2024 及以后。这里我们还需特别强调一下课件与习题课、作业等其它环节的**一致性**。我在做这些东西的时候尽最大可能让一切都保持一致，**甚至包括缩进格式和命名习惯**，这一点既体现在正课和习题课的课件中，也体现在作业、考试、 quiz 的一切示例代码片段和 reference solution 中。**要培养学生良好的代码习惯，就需要首先让他们身处在一个处处都遵循了良好的代码习惯的学习环境中**。

### 作业设计

这一学期只有 7 次作业， 3 次 C 和 4 次 C++ 。原因是多方面的，一是当时的课程和作业进度被国庆放假调休搞得有点乱，二是我同时还要做课件 + CS101 的作业出题，没记错的话到 hw4 前后时课程进度已经到 C++ 了，所以我就从 hw4 开始出 C++ 作业， 7 次作业的占比调整为 8% + 8% + 10% + 10% + 12% + 12% + 12% 。

#### 题目

前三次作业我重新出了一遍，基本都挺正常的。有一些凑数的简单题不谈了。 Spring 2023 的内存泄漏检查器和标准库字符与字符串实现题也原样搬了过来。除此之外我出了个“GCC argument descriptor”，要求学生解读命令行参数，既考察了命令行运行程序如何传参，也强化了对 GCC 常见的参数的含义的理解，也考察了字符串处理，这题不错。

hw4 第一题 Joiner 是写一个简单的类来使用户方便地 join 多个字符串，它解决的问题就是用循环 join 一个 `vector<string>` 里的字符串时总是需要判断当前是不是第一个或者最后一个，这感觉比较烦人。借助这个题我来说说我对此类题目的看法。首先，我不是很喜欢这道题，因为我怀疑它解决的可能是一个伪需求。这个判断并没有真正地被去除，而只是隐藏进了 `Joiner` 类里。它确实让代码变得简洁了，但是它仅仅是节省了一个判断，以及使基于下标的迭代可以被改写为基于范围的迭代而已。学生会感到它很有帮助吗？难说，学生会觉得为了这一点点简洁性却需要多定义一个类。实际上它只有在当前项目里存在大量的 join 行为时，其好处才真正体现出来。但是我还是用了这道题，因为它确实作为一个非常简单的示例展示了人们可能会如何使用 class 。不要觉得这完全没有意义，我确实在 CppCon 某个 lightning talk 以及某个实际的项目里见过这样的东西：

```cpp
template <bool Initial> struct OnceFlag {
   bool value{Initial};
   explicit operator bool() {
      value = !value;
      return !value;
   }
};
```

这其实比那个 `Joiner` 类通用性更强，也更灵活，它没有像 `Joiner` 那样一步到位地让用户省去 `if` 判断，但至少也能让用户可以使用基于范围的迭代。总之，这题依然有其正面的价值，但是如果有更好的题还是把它换掉比较好。

hw4 的第二题我更大胆，压根就没有定主题，直接让学生自己选择主题定义至少两个 class ，要求是有实际意义且使用了一些指定的语言特性，比如 const 成员函数、构造函数等。这题的灵感其实来自《C++ Primer》 7.5.1 节的习题 7.40 ，但不得不说这题比较失败。我的本意是让学生自己尝试用 class 去建模一些自己熟悉的事物，然而此时学生几乎只见过 `Student` 、 `Dynarray` 和 `Joiner` 这三个类，他们根本没有足够的“输入”，让他们强行“输出”还是很为难的。实际上 Spring 2024 的 hw5 和 hw6 （对应到这里的 hw4 和 hw5）我也并不满意，但至少这题给我的教训就是**要让学生多见识一些好的 class** 。或许在这两次作业中，我们应该尝试写一些 class 让学生阅读、分析、修改、使用。

hw5 的第三题我自认为出得不错。它解决的是数独问题，但是其重点在于让学生充分利用我们已经设计好的一个 `Sudoku` 类来实现递归搜索的算法，而且算法本身我们给出了很详细的解释，不像 Spring 2022 那个斗地主那样让学生自学。这题同时也回应了前面提到的一个问题——过度强调类的设计导致学生只会写却不会用。

hw5 的第二题也还行，它是个简单的 STL 算法题，设计了三个场景让学生组合使用 STL 算法来实现，且涵盖了多个不同方面的算法，也包含了向算法传递一个简单的 lambda 。

此外， Spring 2023 有几个题我觉得不错的也搬过来了，比如多项式、表达式树。

hw4-6 依然使用了 Spring 2023 出的那些客观题。我之前说过我觉得这几套客观题不是很好，有点难，但我在这个学期依然用它们，主要是觉得这几次作业的代码量太小了，而当时我确实没有来得及设计更多或者码量稍大一些的题目。尽管我们秋学期的基调是要“降低要求”，但码量这么小还是不太好。未来的秋学期，可以再适当增加一点编程题，同时客观题要用也可以，但是要降低难度，且最好少一点不定项选择。至于什么题型更好，详见 Spring 2024 的 quiz 和考试。

hw7 没有放 OpenGL 游戏，取而代之的是模板的编程题，这是难度显著降低的地方。对春学期的学生来说，模板练不练其实无所谓， OpenGL 游戏这样的大综合题价值更大。而对秋学期的学生来说，写这个大综合有点为难他们了，不如出两个简单有趣的模板题，至少在知识点层面让他们对 C++ 有一个相对完整的认识。可以这么说，在这份作业的题目上，春学期注重深度而秋学期注重广度。具体出了什么题呢？第一题是把 `Dynarray` 类改写为类模板 `Dynarray<T>` ，加上迭代器（就是指针）那一串 `begin`/`end` （包括反向，共 12 个）函数。第二题就有意思了，是一道用模板表达“量纲”的题，来自于《C++ Template Metaprogramming》第三章的示例（有改动），它用一个带有 7 个 `int` 模板参数的类模板来表达量纲，正好对应到物理世界的 7 个基本物理量。同时，这题我们给了学生一定的发挥空间，最后是以代码面谈的方式来评分的，没有做自动化的测试。

看到这里想必你会有一些疑问：上面提到的一些题目好像没法测试，难道全靠人眼看或者代码面谈吗？比如 STL 那题，我们怎么知道学生是不是真的调用了 STL 算法？没错，以前的我顶多只能用一些特殊手段来“hack”，比如把那几个应该会用到的算法为特定数据类型做个特化，记录下有没有被调用。而这学期，我拿出了一个有力的武器。

#### 基于 Clang 的代码检查

Clang 是个好东西。它不仅是个编译器，还把自己的编译器前端作为库设施提供出来，我们可以借此实现自己的工具，做一些代码分析和变换。 LLVM 自己就实现了一系列工具，比如 Clang-Tidy 和 Clang-Query 。 Clang-Tidy 跟我们的需求非常吻合，它就是借助 Clang 的编译器前端来完成各种各样的检测，主要是一些规范性的检测。我们能不能也在 OJ 上部署 Clang-Tidy 呢？理论上完全可行，但实际有一些问题。

首先，我们需要自己实现自己的 checkers （详见下文），而且每道题用的 checkers 都不一样。 Clang-Tidy 的模式是把所有 checkers 都写进 source tree 里，编译的时候一起编译，在命令行通过 `-checks=` 来设置开启哪些、关闭哪些。但是，每次都在那个巨大的 llvm-project source tree 里写东西和编译真是麻烦（当然，也有办法[把 Clang-Tidy 剥离出来](https://github.com/mzyKi/clang-tidy-standalone)），而且我需要的 checkers 和工具其实没有那么复杂，同时也为了在 OJ 上部署方便，加上我那时时间有限需要尽快找到一条可行的路，最终我采用了如下的有些丑陋的方案：

1. 为每道题自己实现一个工具，自己写 `ASTConsumer` 和 `ASTFrontendAction` 的子类，自己调用自己的 checkers ，自己调用 `DiagnosticsEngine` 生成 diagnostics ，只依赖基本的 libTooling 等库，不依赖 Clang-Tidy 框架。如此，每道题的检查器实际上就是一个 cpp 文件，单独编译它即可。
2. 在 OJ 上通过它的包管理器安装了 llvm-15 的库文件。
3. 我们需要链接两个库，一是 libclang （包括 libclangAST.a 、 libclangBasic.a 等一大堆文件，但都是静态库），二是 libLLVM-15.so.1 。于是，我先在本地链接前面那些静态库，然后再放到 OJ 上去链接 OJ 上的 libLLVM-15.so.1 ，得到一个 executable 大约几十 MB ，可以接受，最后把这个 executable 上传为题目测试用的辅助文件。在我的那些文件夹里，你会看到一个名为 `generate_intermediate.sh` 的脚本，它就是我在本地编译的时候采用的指令，最后被注释掉的那行就是我放到 OJ 上去链接 libLLVM-15.so.1 用的指令。
4. 最终把这个 executable 放到 OJ 上使用的时候，还发现它无法正确找到标准库头文件。具体原因没搞清楚，但是我直接把这些头文件的位置硬编码进了 `-I` 的参数里（也许写成 `-isystem` 更好）。在各个调用了检查器的 `compile.sh` 里你会看到那一长串参数。

我当时确实没有把这些搞得很明白，主要的精力都花在了具体的 checkers 实现上。毕竟这已经是一条相对容易操作的路，只是没有那么工程化。有些 checkers 会在不同题目之间共用，我也没有提取出什么可复用的头文件，而是直接复制粘贴。而且，当时我没有足够的经验，我自己都不知道未来我还会需要写什么样的检查器，也就无法知道写成一个什么样的框架或工程是方便的。

这学期我是在 hw4 开始用 Clang 写 checkers 的，后来到 Spring 2024 又补上了前面 C 的作业的一些 checkers ，这里就一并讲讲我到底检查了些什么。
1. 代码规范性和最佳实践的检查。这是最基本的一项，包括 `for` 循环的循环变量要声明在 `for` 语句里、能初始化就不要先声明后赋值、构造函数要尽可能使用初始值列表、某些成员函数应当是 `const` 的等等。这一类的违反，大多数情况下我会生成 error ，也就是说不符合这些规范性或最佳实践的代码直接被视为“编译错误”而没有分数。我其实怀疑这一点可以适当放松，但实际上，我们有不少题目依然是靠代码面谈来检查规范性的，可能是由于给学生自由发挥的空间较大或者某些规范性确实比较难检查。在代码面谈中检查规范性的话，本身就容许学生的代码有一定程度上的违反；而其它那些相对容易检查的题目往往也相对简单，强制要求规范性我认为没什么问题。不过这一点还是要视具体情况而定。另外，对于代码是否格式化，我没有检查，因为不同学生格式化的风格都不一样，而且有些时候轻微的未格式化是无所谓的。我们对格式化的检查都是放在代码面谈环节，而且我们基本只看缩进。
2. 错误提示。比如常见的对 `strcmp` 的误用（`strcmp(...) == 1`），我们会给一个 warning 。再比如，一道题要保存一个多项式的各项系数，题目约定这个多项式的 degree 不超过 50 ，那实际上就需要 51 个位置，我们如果发现学生代码里有 50 的数组，就会给一个 warning 。实践表明，像这样的越界问题，初学者会需要大量的时间才能发现，一是因为他们缺乏这方面的敏感性，二是它很难被发现——学生需要生成恰好 50 阶的多项式来测试，而且就算这么做了，在 Windows 上也往往不会报错。学生的时间是很宝贵的，有些 bug 让他们自己找找当然是必要的，但是像这样的越界 bug 本身也比较无聊，我不认为非得让学生自己找出来不可。再举一个例子，就是那个“GCC argument descriptor”，如果我们发现学生的代码的 `main` 函数不带 `argc, argv` 参数，或者发现学生试图从 stdin 读输入，就会给一个 warning ，因为这说明他肯定没搞明白这题要怎么读那些参数，或者压根就没有听课。这个 warning 里，我们会告诉学生“如果你不知道什么是命令行参数，你应该去翻翻 Lecture 8 的课件”。
3. 题目的特殊限制。有了这种 checkers ，我们就可以检查学生代码的具体实现了，比如有没有用数组、有没有用循环。这种限制究竟做到什么程度要视具体的题目而定，比如 Spring 2024 hw1 的第四题，如果使用了数组或动态分配的内存就无法通过最后的 5 个测试点。注意，我们可以在题目里对实现方式设置限制，但是**必须确保最优解是可以满足这些限制的**，不能为了限制而限制。比如前面提到的那个 STL 算法练习，我们通常都是鼓励能调用标准库就不要自己造轮子的，那道题里设计的情景，除了用 `std::for_each` 输出序列里的元素（这你也不能说不好）之外，确实都应该调用 STL 算法。

不得不说，静态分析就是静态分析，它的静态的属性就决定了我们在写 checkers 的时候必须预判好一切可能的情况。我们在实践中确实遇到了一些先前没想到的情况，使得我们的 checkers 一改再改，也经常在 OJ 上重测。不过我们尽量保证改动都在作业刚发布的几天里做，如果在临近 ddl 时真有什么非改不可的，那就一定要延期 ddl 。有一个我印象很深刻的，就是 [`nullPointerConstant` 这个 AST matcher 的实现是不正确的](https://github.com/llvm/llvm-project/issues/87008)，它判断一个整数字面值 `0` 是不是空指针常量的方法是看它的 parent expression 的类型是不是指针类型，这在类似于 `int *p = 0;` 这种语句里当然没问题（这里 `0` 的 parent 是一个 `ImplicitCastExpr` ，其类型为 `int*`），但是假如 `p` 的类型是 `int *[10]` ，下标表达式 `p[0]` 的类型也是指针！这就使得我们的 checker 给出了“应当把 `p[0]` 改写为 `p[NULL]`”的荒唐建议。

这些基于 Clang 的 checkers 在 Fall 2023 和 Spring 2024 的作业中的确发挥了很大的作用，帮助我们完成了很多的检测，使我们可以出更多样的作业题目，也使作业评测的反馈更加丰富。但是，开发这些 checkers 是有一定难度的，一是在于学习 Clang 前端这套东西，二是在于设计规则时要考虑得非常周全。举个例子，当你看到学生写出 `some_condition ? true : false` 时，你会希望 checker 能识别出来并建议学生直接写 `some_condition` 。但是， `some_condition ? 1 : 0` 你管还是不管呢？如果只是在语法树层面进行匹配，你怎么知道这里的 `1` 和 `0` 是被当成了 True 和 False 使用，而不是真的使用这两个整数呢？应该建议学生把 `int initial_value = some_condition ? 1 : 0;` 改为 `int initial_value = some_condition;` 或者 `int initial_value = (bool)some_condition;` 吗？我不赞成。目前的 AI 的能力其实可以很快地教会你 Clang 前端的基础知识，甚至能直接帮你写 checker ，但是设计一个适用于语法树层面的匹配的合理的检查规则是需要非常谨慎的思考，甚至可能需要语言律师的，要让 AI 来完成这方面的工作是需要一些调教的。

语法树层面的检查还有一个重要的问题，就是模板处理起来有些麻烦。众所周知， C++ 编译器如果只是看到模板本身，它只会做一些基本的 parsing ，在语法树上你会看到大量诸如 `DependentTemplateName` 这样的结点，子树里啥也没有。只有当模板被实例化出来，其中的代码才会被进一步地解析。所以如果要检查模板代码，那就需要专门写一个文件去显式地实例化这个模板，然后对着这个文件检查。

AI 时代，能不能直接用 AI 来做这些检测？我没试过，不知道它的一致性和成本如何，但我觉得应该这么做，或者至少是 AI 与 Clang checkers 结合。

### quiz 和考试

这一学期的 quiz 1 和考试，题型都和 Spring 2023 差不多，而难度显著降低。考试的选择题里应该只有一道比较 tricky 的，就是第 11 题的 D 选项，需要学生意识到把 `new` 改写成 `std::make_unique` 就无法调用 private constructor 了。这个考点其实我并不喜欢，它本身就是 C++ 设计得不好的地方，不过作为一个选项用来稍微提高一点点难度其实也还说得过去，也并非完全判断不出来。我不喜欢把那些应对语言设计本身的缺陷而产生的 workarounds 当成要求学生掌握的知识点，比如关于为什么在 C 里不要给 `malloc` 的结果加显式转换，我们在课上只讲“因为这个显式转换没有必要”这一个原因，而实际上真正的原因是 [C 非常愚蠢的“假定 `int`”的历史遗留问题](https://stackoverflow.com/a/7545394/8395081)。

我们允许携带一切纸质资料。我曾设想过更加大胆的开卷方式和考试形式，比如允许带电脑使用编译器和查阅本地资料，甚至于直接允许上网。这些是合理的，哪有人编程的时候不被允许查资料的？但是，我同样反对认为闭卷考试一无是处的看法。闭卷考试的主要作用是考察对最基本的知识的熟练程度，以及对知识点的宏观认识。比方说，实际编程的时候突然忘了 `std::string::find` 成员函数的具体用法，我们可以上网现查或者问 AI ，这很正常；但如果你连怎么创建一个 `std::string` 都不知道，你当然仍然可以上网查或者问 AI ，但这种问题就过于初级了。如果什么都要从这种最初级的问题开始查或问 AI ，那开发效率就会非常低下，而且这暗示你对于这门语言提供了什么东西也缺乏宏观的认识——你就不可能有真正解决问题的能力而全都得靠网络或 AI ，因为你连自己手头有什么工具都不清楚。因此，一个考试如果是闭卷的，其重点就应放在这类基本知识和宏观认识上；一个考试越是开卷，就越是可以加入开放性的、实践性的、综合性的问题。但同时，基本知识和宏观认识的考察依然要有，不论是何种程度的开卷，因为我们是一门课程的考试而非选拔性竞赛。大学课程的考试应该是合格性的，困难的部分应该放到更切合实际的、实践性的部分去，比如大作业、论文研讨。此外，我非常不喜欢只能携带一两张 cheat sheet 的半开卷，因为它实际上限制了学生的复习方式，而且做 cheat sheet 是比较耗时的。不是所有学生都喜欢动笔记下很多东西的，重要的不是你写下来了多少东西，而是记到脑子里、悟到了多少东西。我自己遇到这种半开卷的考试，几乎都是自己复习完了之后问别人复印一份 cheat sheet 带上考场。

题目方面，我开始尝试出一些更开放的题，比如我直接出了一道简答题，要求学生阐述“is-a”关系的含义，并举出正例和反例。其实就是对 lecture 22 前半段或者说《Effective C++》条款 32 的总结。 Quiz 2 安排在 Lecture 23 （More on STL）之后的两课时，当时老师正好有事，就直接在那天的两课时安排了一次 quiz 。在 Quiz 2 里我做了更多的尝试，出了三道简答题，这里仔细说说：

1. 第一题涉及 `std::map` 的基本使用。第一小题问的是 `std::map` 的 `operator[]` 在创建新元素时会如何初始化，这实际上是我们在讲 STL 的时候强调过的一个思想：标准库的设计者和我们一样讨厌未定义的值，因此在能避免未定义的值的地方它总是会尽力避免，比如所有涉及到“默认地”构造一个对象的地方它都会采用 value-initialization 。当然，课上是讲了 `operator[]` 会采用 value-initialization 的，但如果你忘记了，只要你记得这一核心思想，一样能答得上来。第二小题则问 `operator[]` 有没有 `const` 版本，这其实是课上没有明确地提及的，需要学生经过一定的思考。这个 `operator[]` 没有 `const` 版本，是因为它在当前 key 不存在时会创建元素，这修改了容器的状态，而对 `const` 容器的按 key 访问有 `at` 帮忙；如果提供一个不创建元素的 `const` 版的 `operator[]` ，其语义和 non-`const` 版本不同，这可能会造成误解。当然，我感觉这个有些难了。
2. 第二题是设计个继承方案并解释原因，其实主要就是考察对“is-a”关系的理解，不是很难。这属于一个比较基本的题目，需要学生按部就班地把一个类的代码写出来，继承关系设计也不难，原因解释也不难，属于给辛苦分。顺带提一下，手写代码里出现小的编译错误，我强烈反对“突然死亡法”（你的程序不能编译，所以一分都没有），而是赞成类似于高考作文错别字的处理方式。
3. 第三题引入了一个课上没有讲过的语言特性（实际上出现在了之后的作业里，即表达式树那题）——用 `using` 继承构造函数。这题和第一题放在一起就充分地体现了我一贯的观点：一个 C++ 高手并不是熟记所有语言特性的每一个细节，但是当他忘记某个特性的细节时，他能够根据自己对 C++ 的理解推断出那个特性会是什么样子。这里的第一小题是问继承而来的构造函数的访问权限是什么，那当然是跟父类里的一致了——不然我岂不是可以通过这种方式将父类的私有构造函数暴露给外部？继承关系必须尊重父类的封装，比如子类的构造函数必须调用父类的构造函数来构造父类的部分，而不能直接初始化父类的数据成员，这个思想是我们在课上强调了的。第二小题问的是由 `using` 继承而来的构造函数会如何初始化子类自己的数据成员。这也是很好推断的——顺序是在父类子对象初始化完毕以后按声明顺序，方式是有类内初始值则用类内初始值，无类内初始值则默认初始化。它其实就相当于一个没有在初始化列表里写出数据成员初始化方式的构造函数。

最后还有一道关于重载运算符调用版本的判断，回应了那段时间答疑的时候发现的问题。总的来说这个 quiz 的题目挺有意思的，特别是那个 `using` 继承构造函数题。注意，这场 quiz 是两课时的，学生有充分的时间写这么多字。

这些给 Spring 2024 的 quiz 和考试的进一步的改变奠定了基础。

## Spring 2024 ：最终的方案

Spring 2024 是我做 CS100 助教的最后一个学期。有了前面足足三个学期的经验和准备，我们的“最终方案”基本成型了，尤其是把课件和习题课课件都改到了比较令人满意的程度，终于不用再一周一周地做课件了。课程中的大部分环节都可以照搬之前的，不过还有两个部分我们需要继续改进：

1. 作业要继续出一些新题目，想一些新形式，并全面应用 Clang checkers 和代码面谈。
2. 结合先前的经验和思考，重新设计 quiz 和考试。

另外不要忘了，习题课的课件总是需要针对实际进度和学生的情况进行微调的。

### 作业

宏观上讲，作业的整体风格没有特别明显的转向，但在具体的题目和形式上也有不少改进。我们更多使用代码面谈来评分，在 hw4 、 hw6 和 hw8 中我们都使用了代码面谈。我们全面应用了 Clang checkers ，并且更充分地发挥 OJ 的能力，设计了一些更复杂的评分机制并借助 `compile.sh` 和 `execute.sh` 实现（事实上这两个脚本配合几乎能实现任何评分机制）。前三次作业设置了客观题，而后面 C++ 部分不再设置客观题。此外，第四次作业也做成了一个小型 project ，是一个完整的、可以玩的命令行 Pacman 游戏。

在刚起步的阶段，我们感觉到有许多非常重要的基础知识不太容易塞进前三次作业总共十几道的编程题里，有些知识点可能比较零散而不容易出编程题，或者有些知识点其实是理解性的、概念性的，编程题的形式不太适合，不如直接以客观题的形式考察，因此我们在前三次作业中加入了客观题。但是这些客观题的难度和题量都显著低于之前两个学期的 C++ 作业里的客观题。客观题要注意目的性，要想清楚不同的知识点应该掌握到什么程度，并有针对性地选择合适的题型，要避免无意义的、刻意构造的场景设计，避免刻意刁难学生的脑筋急转弯题目。同时，作业题相比于考试题可以更细一些，可以略微更“语言律师”一点点，因为做作业的时候学生有充足的时间和一切参考资料。

这一学期的 C++ 作业没有设置客观题，主要是因为我们感到 C++ 部分的内容确实较多，想要尽可能地精简作业题目，优先保证那些最实用的内容。如果能设计出几份好的客观题，放到 C++ 作业里也是可以的。

前三次作业除了增加客观题之外，编程题基本上复用了之前的题目，或者沿用了之前的设计理念。值得注意的是，我们结合 Clang checkers 和 OJ 的能力实现了更复杂的评分机制，比如 hw1 的第四题，如果 Clang checkers 发现提交的代码中使用了数组，我们就直接将最后五个 testcases 屏蔽，而不是想办法设计这些 testcases 让它无法通过。 hw3 的第二题是一个小小的尝试，它的第一个 subtask 是要求学生写一个判断回文的程序，而第二个 subtask 要求学生给第一个 subtask 写一个测试数据生成器，可以用 C 也可以用 Python 。像这样的题目，一种简单的测试方法就是在 OJ 上开两个题，两个提交入口分别提交两个 subtask 的程序，不过我们当时做得稍微有点复杂（其实也是有点想试试这个 OJ 能怎么玩），我们让学生把两个程序打包成 zip 上传，充分利用 `compile.sh` 和 `execute.sh` ，有一部分测试其实是在“编译时”即 `compile.sh` 里完成的。为了在提交界面仍然能显示出学生提交的代码，我们还用 `pygmentize` 把代码带有语法高亮地打印出来。

hw4 我们出了一个命令行字符版 Pacman 小游戏，涉及到输出颜色、光标设置等部分的功能由我们来提供，基本的框架也是我们搭好，而学生要补充其中核心的部分。我们也提供了一个实现好的可执行程序作为功能的参考。这道题我们完全采用代码面谈的方式来评分，面谈的内容包括对代码的理解、功能完成度演示和代码质量。具体来说：
1. 代码理解方面，我们主要问两个点，一是两个与 `enum` 相关的函数的实现原理（这两个函数是我们在框架里给出的），二是关于 `createGame` 的设计。我们在框架里给出那两个`enum` 相关的函数，有效地引导了学生利用它们写出简洁的代码，而不是针对每一种移动方向把一模一样的代码重复 4 遍甚至 8 遍（这在 Spring 2022 的 Pacman 提交里非常常见）。此外，关于 `createGame` 的设计的讨论是在题目描述里提到了的，所以在代码面谈时问这个点也是非常合理的。在设计代码面谈里的问题时一定要想清楚学生到底能不能想到这个问题，有些是需要加以引导的。
2. 功能完成度方面，我们吸收了 Spring 2023 的植物大战僵尸的经验，只要这个游戏玩起来没问题就可以，不去尝试编写测试。我并不是完全否定通过测试代码来测试功能，只是对于这类作业题似乎确实不合适。
3. 代码质量方面，我们采用扣分制，类似于考驾照科目二和科目三那种计分方式。特别是我们规定代码必须无 warning 且格式化，这两点违反任何一个都会扣 50 。不过考虑到这是第一次，我们实际在面谈时允许学生当场改正，如果能把 warning 修掉或者知道如何格式化就不扣分，不知道如何格式化的我们也会现场教（实际上我们在 Lecture 3 slides 里就有讲如何格式化）。对于格式化的判断，我们主要看缩进，只要缩进正确即可。还有一点值得一提，我们要求学生“尽可能”使用 initializer-lists 或 compound literals 来对 `struct` 对象进行赋值或初始化，这里的要求说的是“尽可能”，而实际检查的时候只要至少使用了一次就不扣分。

读者可能会问，为什么代码质量不用 Clang checkers 来检查呢？确实，不仅这一题我们没有用 Clang checkers ，稍后会讲的 hw6 和 hw8 也没有。究其原因，还是编写这些 checkers 耗费的时间有点多。很多规则其实在代码里扫一眼就能知道遵守与否，但要写 checker 则需要从规则设计到 checker 实现都要保证严谨，像“避免使用 magic numbers”和“使用有意义的变量名”这样的规则要写出合适的 checker 其实挺困难的。确实，**Clang checkers 其实也不是完美的方案**，种种事实和经验都呼唤着 LLM 。

hw5 一共两道题，第二题是常规的 `Dynarray` 没什么问题，而第一题又是一个烂活，依然是 Fall 2023 “Design Thinking”那一类的题目，我们给了几个主题让学生定义一些类来建模，要求用到我们讲过的一些特性，比如成员函数、构造函数。但事实表明学生根本不知道这些类该怎么设计，比如哪些地方该用引用、哪些地方该拷贝，应该提供哪些构造函数、哪些 getter/setter 等等。两次经验都表明在这个阶段让学生自由作画是完全错误的，因为这时学生刚刚踏入 C++ 的大门，**根本没见过好的类设计是什么样的**。所以说，**在这个阶段应该出一些代码阅读理解**，带着学生看几个好的类设计，引导学生思考这些类为什么这样设计，设计者打算让用户如何使用这些类。实际评分的时候，提交的程序只要包含了我们要求的那些特性就给满分，不设代码面谈，但学生如果对自己的设计有什么疑问可以找 TA 寻求指导。

hw6 的前两题是 `Dynarray` 和 STL 算法练习，第三题我们重点说说。这道题我们的本意是想让学生体验一下代码重构——如何重构能使得代码更模块化、更易于复用和扩展，这个想法其实来自于《C++ 沉思录》第 9 和第 10 章。然而，我们最终选择的这个比赛记分牌的例子其实也并不好，重构带来的效果并不明显，而重构本身所增加的代码量其实是会带来学生的质疑的。究其原因，是这个记分牌确实没有太多扩展的可能，学生无法直观地感受到重构带来的灵活性的增加。希望未来能有更好的素材。

hw6 有代码面谈，一是 `Dynarray` 有一部分面谈的问题（这是从 Spring 2023 开始就一直有的），二是第三题我们也纯靠代码面谈来评分。这里有一个重大失误，就是我们题目描述中的“扩展记分牌的功能”一节给出的两种“可能的扩展”实际上是**递进关系**，但我们没有说清楚。实际在检查的时候，我班里有一名同学实现了 2 但没有实现 1 ，即他把分数上下放置了但没有增加“局分”。幸好这样的人确实不多。

从这一次代码面谈开始，我们不再允许学生现场改正有 warning 或未格式化的代码，该怎么扣分就怎么扣分。

hw7 的第一题依然是 Fall 2023 那个表达式树，第二题则是在此基础上，考虑程序语言中的表达式而非数学表达式，要求使用 Visitor Pattern 对表达式进行类型检查。这道题很有意思，它最终要实现的东西其实是很简单的，而重点是在“拓展眼界”上—— Visitor Pattern 是一种设计模式，类型检查则是在编译器视角的一个事情，这两个都是课上没有讲过的。我总觉得编程课或者软件工程课应该讲一讲设计模式，它理解起来其实没多难。不过在今天，设计模式的地位的确有点尴尬。设计模式越来越不需要借助传统的方式（即通过类和继承）实现，即便是讨厌函数指针的人也会使用 `std::function` 、 lambda 、 模板等方式来实现，很多时候我们在不知不觉中就已经实现了某个设计模式。尤其是在 AI 时代，设计模式是否确实应该让位于其它的一些更重要的技能？

另外， hw7 的第二题我们故意引入了两个 translation units ，以此为例子演示了一下多文件编译，为 hw8 做铺垫。多文件编译到底何时讲、怎样讲，要不要提到正课讲，我觉得是需要好好考虑一下的。

hw8 沿用了 Spring 2023 的植物大战僵尸，但我们采用本学期 hw4 和 hw6 的模式，完全靠代码面谈来评分。不再需要走后门跑测试，学生也就有了更大的自由度，可以随便改框架里的代码。而和 Spring 2023 不同的是，这一次我在技术层面亲自接手了这道题，还是发现了不少问题的。

首先是其他的助教（我忘了具体是谁）发现，只需在 `Framework` 的 `CMakeLists.txt` 里把 `SOIL` 和 `freeglut` 的链接顺序调整一下，确保 `freeglut` 在 `SOIL` 之前，即可让 MinGW 正常编译。因此，这道题实际上完全不需要安装 Visual Studio 。第二是关于文件结构，在 Spring 2023 我们是把头文件和 `.cpp` 文件放在一起的，这样我们只需直接替换整个 `Framework` 文件夹即可走后门跑测试；而在不需要跑测试的情况下，分 `include` 和 `src` 两个文件夹其实是更合理也更方便的结构。第三， Spring 2023 的版本的 framework 中对于智能指针的使用是比较保守的——全部采用 `std::shared_ptr` ，而实际上仔细思考对象之间的关系就会发现 `std::shared_ptr` 是完全不需要的，全部可以用 `std::unique_ptr` + 裸指针（代表“observer pointer”）实现。此外， Spring 2023 的框架里对于究竟什么函数应该放在 `.cpp` 里定义也没有想得很清楚，像一些默认行为的析构函数和 getter/setter 这类非常简单的函数完全可以直接定义在头文件里的类定义内部。只有那些体积很大或者存在依赖问题的函数才需要放到 `.cpp` 里定义。

遗憾的是，由于时间紧迫，我们在发现以上问题之前就已经把作业的材料发给学生了！这导致在那几天里我们不断地更新题目文件，新的文档与旧的文档并存，极其混乱，学生被迫不断地从 Piazza 下载新的文件。所以说，这道题的材料**缺乏彻底的整理和更新**，这是一个非常致命、必须解决的问题。

### quiz 和考试

这学期的 quiz 是一次，在差不多刚讲完 C 、开始往 C++ 过渡的时候。这次 quiz 出了两道题：
1. 考虑一个名为 `remove_digits` 的函数，它的作用应当是“删除”一个字符串里所有的数字字符。在实际程序中，这样的函数的接口设计和实现方式都会有很多种可能。接口方面，究竟是在原地把数字字符删掉，还是生成一个新的、不包含数字字符的字符串，还是把数字字符挪到串的末尾去？如果生成新的字符串，应该以什么方式“返回”？实现方面，下标的计算、内存的分配和使用也都是容易犯错的点，而且需要与其接口设计相符。我们给出了 4 个版本的 `remove_digits` 函数，每个版本都在函数前面加上了文档注释，写清楚了这个函数做什么、参数的意义是什么等等。要求学生判断每一个版本是否正确，如果错误的话是错在了哪。这道题好在以下几点：
   - 它取材于一个足够简单、十分常见且确实有好几种不同实现方式的问题。
   - 它要求学生阅读代码中的文档注释、函数声明和函数实现，要求理解三者之间的关系，特别是读懂文档注释中暗含的约束，或者说“契约”。比如， (b) 选项的文档注释中声称“`dest` points to a block of memory large enough to hold the result”，而实现中执行了 `strcpy(dest, source)` ，这实际上要求 `dest` 指向的内存得至少和 `source` 一样大，而不是只要能存下结果就行，这就是不匹配。
   - 它也考察具体的实现细节，比如有一个选项涉及一个下标计算错误。
   - 学生要作出正确/错误的判断，并写出理由，而不是仅仅选出一个正确的选项。在 quiz 中，这个判断和理由各占 50% 的分值。如果这是考试题，可以把判断的分值占比缩小到 20% 或者更低。
2. 定义一个函数 `length_sum` ，接受一个 `std::vector<std::string>` ，用基于范围的 `for` 语句求出其中所有字符串的长度的总和并返回。其中，我们抠出三个部分让学生填空，一是参数的声明，二是 `for` 语句，三是 `sum +=` 后面的部分。这也是一个非常简单且有代表性的例子，而且只考察了最基本的 pass-by-reference-to-const 和基于范围的 `for` 语句。

Quiz 和考试既有考察的目的，也有沟通的目的——它们让我们在特定的时间节点通过一份题目来告诉学生我们认为什么是重要的、哪些内容应该掌握到什么程度。就算我们主观上没有从这个角度出发来出题，客观上它也会向学生传达这样的信息。当然，这本身也是我们课程设计的一个弱点，如果能把不同的内容分别要掌握到什么程度说得更明确一些就更好了。

你可能已经发现，从 Fall 2023 的 quiz2 开始，我就不再使用选择题了。这学期的 quiz 由简答题和填空题组成，考试也会是这样。我抛弃选择题并不是出于对老江苏高考数学的什么特殊情结，而是因为选择题有至少以下几个缺点：
1. 易受考试策略影响，不容易反映学生的真实掌握情况。对于单选题，你确定一个选项是对的，其它就不用看了；对于不定项，有时你保守地只选择最稳妥的选项，却能得到比掌握得实际上比你更好的学生更多的分数。大学课程的考试和高考是不一样的，大家参加高考时都是做了充足的准备的，这种保守策略的普遍程度远低于大学考试。当然，我们针对不定项选择可以采用更精细的打分方式，就是阅卷稍微麻烦点。
2. 选择题对于知识点的考查方式不够直接。有些选择题的四个选项是对同一个问题的四个答案，要学生选出正确的答案，那为何不直接让学生写出正确的答案呢？有些选择题实际上就是四道判断，四个选项看似围绕同一个程序或话题展开，实际上说的是四个不同的事情，我们出题得想办法找一个好的由头把它们捏在一起，学生做题也很难受，无形中也增加了题量。

另外，我们也发现很多学生不会使用专业的术语进行表达，这本质上还是没有真正学懂、学透，还是浮光掠影，看到诸如 `int x;` 这样的局部变量声明，可能隐约知道 `x` 的值是有问题的，但具体是什么问题呢？“局部变量未初始化”、“未定义的值”这样的术语却死活说不出口。以往我们的考试题都是以选择题为主，后面的非选择题要学生写的也主要是代码，而不是文字。我觉得应该要有简答题。一个代码片段到底有什么问题，不该只是提供几个可能的运行结果让学生来选，完全可以让学生自己写一段分析。

考试方面，我们采用填空题+简答题的形式，题目更加接近实际，场景设置更加简单清晰。可以这么说，我们的出题思路**从对知识点和概念进行抽样，转向了对具有代表性的使用场景进行抽样**。这里我们先看几个具体的填空题的例子。
1. 第三题，给出了一个 `struct` ，问如何用 `malloc` 分配一片内存来存放 `n` 个对象。非常简单直接，就是 `n * sizeof(struct XXX)` 。
2. 第四题，给出两个指针 `p1` 和 `p2` 分别指向一个数组中的两个位置，问 `p1 - p2` 的类型是整数还是指针，以及其值是什么。我们甚至不要求写出 `ptrdiff_t` 。
3. 第五题，给定一个 `std::string` 对象 `s` ，问如何用 `new` 创建一个新的 `std::string` 并且从 `s` 移动构造。这个 `new` 表达式被用来初始化一个用 `auto` 声明的变量 `t` ，问 `t` 的类型，以及如何销毁这个对象并释放内存。
4. 第六题的前两空，我们考查最简单的 `push_back` 和基于范围的 `for` 语句的使用，第一空需要注意加 `std::move` ，第二空需要注意写 reference-to-`const` ，两个细节就把那些真学懂了的和只是听说过的学生区分开了。第三空更有意思，我们挑了一个课上没有讲过的 STL 算法 `std::count_if` ，给了一句类似于“接受一个序列和一个一元谓词”这样的简单描述，然后让学生尝试写出如何使用这个函数来实现某个目的。我们在课上讲 STL 算法时就强调过，要注意这些算法的普遍规范，抓住它们在命名、传参、行为方面的规律，而不是一个一个地去记。
5. 第七题，给出了一个 `class Matrix` 的简单实现，重载了 `operator()` 来实现二维下标访问，并且给出了一元负号 `operator-` （`friend`）、二元加号 `operator+` 和 `operator+=` 的声明。这里我们不要求学生写出这些重载运算符的实现，而是考查学生对这三个声明的理解。我们首先让学生给这三个声明正确地加上 `const` 修饰符，这就得认识到 `friend` 不是成员函数，以及 `operator+` 和 `operator+=` 分别是否改变 `*this` 自己。然后，我们让学生写出一个调用了上述 `operator-` 的表达式，这就涉及到理解这个 `operator-` 到底是一元负号还是二元减号，以及理解重载运算符到底提供了怎样的便利。第三小问，我们让学生写出二维下标访问的表达式，这就需要看懂这个 `operator()` 。你应该感受到了，我们的重点完全放在了对已有的重载运算符代码的阅读理解与使用上。大多数时候我们都是在使用其它程序库所提供的重载运算符，而不是自己写。就算真要自己写，这个运算符重载本身的实现也往往是 trivial 的，真正的重点依然在如何让用户用起来更方便上。

后面的三道简答题也非常有意思。
1. 第一题，依然回归 `Dynarray` 这个例子。我们首先介绍一个课上没有讲过的语言特性：委托构造函数。我们给出两个委托构造函数的例子，让学生尝试把另一个构造函数也改写成委托的形式，这要求学生看懂一个没见过的语言特性，建立它与自己所学过的知识的联系，同时“委托”本质是要理解各个构造函数之间的关系。第二小问，我们拿出经典的 copy-and-swap idiom ，问那个 by-value 传参的赋值运算符是什么赋值运算符，要写出原因。第三小问，我们要求学生为 `Dynarray` 写出符合规范的 `begin()` 和 `end()` 函数，这实际上同时也考查了 STL 容器和迭代器。总的来说，这道大题就是在大家无比熟悉的 `Dynarray` 例子的基础上进行拓展和延伸。
2. 第二题是我的一个尝试。我挑了两个场景，给出代码片段和编译器的报错信息，让学生来解释编译器报的错是什么意思，以及这个代码应该怎么改。这是很现实的，就是编译器的报错有时并不直接对应到代码中的错误。第一个场景中，有一个函数以 by-value 的方式接受一个 `std::unique_ptr` 参数，调用者传了一个左值的 `std::unique_ptr` 进来，触发了“use of deleted function ...”的报错，问编译器提到的那个“deleted function”是什么，以及这个代码应该如何修改。第二个场景涉及到构造函数初始值列表没有按照成员声明顺序书写，且先初始化的成员的初始值依赖于后初始化的成员，这时编译器给了一串 warning 。我自己真的在这个问题上栽过跟头，多年过去依然记忆犹新。这个场景也能进一步让学生认识到 warning 的重要性。
3. 最后一题考查继承和多态，我的考法非常简单，就是围绕《Effective C++》条款 34 ，构造了一组和书上几乎一样的例子，只是换了个皮。一方面这个场景确实很典型，另一方面我还是不想在继承和多态上考得过深、过细，还是要抓住最关键的知识点。

我对这份试卷是非常满意的，没有刻意构造的复杂场景，没有连篇累牍的案例分析，没有死抠字眼的概念辨析。几乎每一题都需要手写一点点代码，但又不同于刻板印象中的“手写代码”题。不涉及什么无意义的考试策略，会就是会、不会就是不会。题量适中，时间对学生来说基本足够，中等偏上水平的学生一个多小时即可做完。

书面考试和 quiz 的终极形态大概就是如此了。注意我加了一个定语：“书面”。未来的考试和 quiz 是否可以以及是否有必要结合机试、面试等更丰富的形式呢？这也需要结合未来的课程内容及教学目标的调整来判断。

### 问题

在 Spring 2023 的部分我们曾说到， hw3 前后有部分同学抱怨作业量大。在 Spring 2024 也发生了同样的事情，有两位同学在 Piazza 上发帖抱怨作业太多/太难。在我看来这个问题有以下几个方面（不分先后）。

1. 初学者有些人上手偏慢，需要适应，需要指导。客观来说，我们前三次作业的难度和题量肯定是不大的，如果让当时发帖的两位学生在学期结束后再回头看当时的题目，他们一定也会得出这个结论。但是，大家普遍都没怎么接触过编程，在起步阶段学生要学习的不仅仅是一些语言知识，更重要的是写程序所需的那种思维方式。所以，我们要做的应该是优化学生的体验，包括下放测试用例、主动提供更多的指导。前面已经说过，这一阶段的简单题和中档题的练习是必不可少的，盲目砍作业量不可取。
2. 学校的培养方案让某些非 CS 专业的学生必修这门课，甚至包括一些创艺学院的学生，这一点我觉得是值得商榷的，或者也许让部分学生单独排名给分会好一点？而其实 CS 专业本身的培养方案也有优化的空间，这一点我放到后面讲。
3. 我们的课程内容确实太多太多，已经到了必须删减的时候。我们在这几个学期对课程进行改革的时候，确实轻视了这一点，即学生的精力是有限的，并不是只要一个知识点足够简单或者用足够好的方式教给学生就一定能被接受。我们当然不是故意忽视这一点，一方面我们当时的主要精力都在提升内容质量上，另一方面我们的确缺乏及时的、全面的调查。可能是没有足够的精力，也可能是单纯的畏惧，亦或者是不知道怎样调查，总之在那个时候我没有对学生对知识点的掌握情况进行充分的调查。

学生在作业中对 AI 的依赖也在这一学期逐渐演变成一个不可忽视的问题。从三次代码面谈和期中考试的结果来看，目前我们的模式对于学生的学习效果还是有一定保障的，主要是分梯度的题目、丰富的作业形式和代码面谈在起作用。但是考虑到课程内容的量和现在 AI 飞快的发展速度，这种状态其实是相当脆弱的。

随着 AI 的全面普及，我们在教学中究竟应该注重什么、弱化什么都在发生着深刻的变化。在 Spring 2024 刚结束的时候我认为，如果再给我一个学期，我一定会对课程内容做减法，增加更多更通用的、基础的技能的培养，比如文档的阅读和撰写、工具的使用。现在，一个全面的改革已经迫在眉睫。

## 接下来怎么办？

我所参与的工作正常来说到这里就该结束了。然而到 Fall 2024 ，出于一些原因，我也帮助当时的助教简单改了下期中考试的题目。我感到有些事情不能只停留在我自己的脑子里，所以写了这篇长文。可能这种按时间顺序组织文章的方式并不是最好的，但写出来就比不写出来好，况且现在有 AI 辅助阅读。

关于“接下来怎么办”，我大致分两类来说一说。

一是我们之前就应该做而没做或没做好的，包括我注意到某些助教没有做好的。首先，必须注意一切内容的规范性问题，包括英语语法和标点符号，以及专业术语的使用。我们作为教学团队的成员，我们所产出的内容必须首先成为学生的榜样，要体现出我们的专业性和严谨性。英语说不清楚的事情可以用中文说，就像我之前有一些作业题是用中文写的一样。现在有 AI 的情况下，做到这一点应该越来越容易了，但也要注意对 AI 生成的内容的监督，特别是细节上的一些用词问题。第二，有一些事宜始终没有很好的文本性说明，可能每次都停留在口头，在习题课或者在私下与学生的聊天中交代，一个重要的例子就是 OJ 的使用。每个学期都有同学在一些多组输入输出的题目上出现疑惑，他们以为需要把所有的输出内容都存下来，待到全部输入结束以后再输出。而且，随着我们 OJ 玩得越来越复杂， OJ 给出的反馈也早已超出传统的 AC/WA/RE/... 我们需要有一份文本来把这些事说清楚，特别是要明确哪些是编译器的诊断信息、哪些是我们自己的 checkers 给出的反馈。最后，也是前面提到过的，要眼睛向下，对学生的学习状况进行充分的调查，调查要是全面的、及时的，不是调查某几个和你关系近的学生的情况，而是调查各个层次、各个专业的学生的全部的情况。有了充分的调查，你才知道你所做的事情对不对，学生是不是像你所设想的那样学到了东西，而不是主观主义，“跟着感觉走”。但我们也要注意正确地看待学生的反馈，曾经有些助教极其迷信学生基于自发的感受给出的意见，见到一点点抱怨就盲目否定我们的工作，这是不对的。

二是之前没有条件做而接下来应该做的。最重要的当然是课程内容的转向， Fall 2025 似乎开了一个不错的头。我前面这些工作的总体方向就是让学生把这两门语言本身学得更好，但其实像文档的阅读和撰写、工具的使用、调试的技巧、对系统的认识等等这些更加通用的基础技能也是同样重要的，而 AI 的普及和发展又对这个天平带来了更大的改变。可能有些助教感到自己的 C++ 水平不足而不敢动我制作的课件，其实大可不必如此谨慎，尤其是现在有 AI 帮助的情况下，确定某些内容可以删减就大胆去做吧。实践是检验真理的唯一标准，而不是其它任何的教条。同时，最好还能对每部分内容究竟要求学生掌握到什么程度有一个大致的说明，不需要具体到知识点，但总比全让学生自己猜要强。我前面说我们通过作业、 quiz 、考试隐式地告诉学生我们认为哪些是重要的，但如果能显式地把这件事说出来，学生的学习目标也会更清晰一些。最后，我们的技术也要与时俱进。之前我加入的各种代码检查和 Clang checkers 都是对简单的运行时测试的不足给出的弥补，但现在我们也看到 Clang checkers 的不足，而 LLM 在处理我们这门课的作业代码这种简单的、小规模的代码上已经具有相当好的表现了。相比于让 LLM 帮忙写 checker ，我们可以直接用 LLM 来检查代码，不仅可以用 LLM 检查代码规范性，还可以检查功能和逻辑。而且我相信随着 LLM 的快速发展，未来我们还能更充分地利用 LLM 设计出更好的课程形式，这里我就不做过多的想象了。

前面所说的一切，都是限定在 CS100 这一门课程内部我们做了什么、还能做什么。但如果进一步地，想一想 CS 学生修读的整套专业基础课程，其实我希望几门课之间能有更加紧密的联系，尤其是和大二上的 CS101 （算法与数据结构）。 CS101 的大多数知识点无论是理论还是代码实现都并不难，它们对于培养学生的编程思维也是大有裨益的，如果能和 CS100 并行地、穿插着学，也许会有更好的效果。当然，我不是说要学生在一个学期里同时学完 CS100 和 CS101 ，我觉得大可以把它们合并成一门两学期乃至三学期的课程，这在一些高校是有实践的。

## 结语

上科大提供了一片非常好的土壤，可以让学生亲自来对课程开刀。而我能有幸参与 CS100 一连四个学期的迭代，这的确是很难得的、很独特的经历。可能最初我只是想借助这个岗位展示和锻炼自己，但参与得越深，越感到责任重大。今天把我当时的所作所为、所思所想完整地记录下来，不是自信我做得有多么好，而是为这些工作留一个记录。后人如果想要就某些部分进行讨论，通过本文也就能看到我当时的思考。

计算机是飞速发展的学科，特别是当下正值 AI 的大潮汹涌，想要培养出真正适应这个时代的学生，就需要教学团队不断地进行改革。坚持对知识、对学术严谨认真的态度，对自己在工作中产出的一切负责，处理各种问题时坚持实事求是、以实践为标准，剩下的就是大胆去想，大胆去做。我们所做的事肯定不会十全十美，“但历史毕竟是历史，我们完全可以了无遗憾”（《摆脱贫困》之《跋》）。
